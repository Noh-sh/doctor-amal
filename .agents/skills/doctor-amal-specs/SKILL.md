# Doctor Amal Specs

Этот skill является навигатором по спецификациям проекта Doctor Amal. Он не заменяет `spec/` и не копирует требования внутрь себя. Источник правды всегда находится в `spec/`.

## Обязательный порядок работы

1. Сначала прочитать `AGENTS.md`.
2. Определить тип задачи пользователя.
3. Открыть релевантные файлы из `spec/` по карте в `references/spec-map.md`.
4. Для изменений поведения и требований свериться с `spec/technical-specs/change-management.md`.
5. Найти подтверждение требования в specs. Если это технический или дизайнерский вывод из specs, явно отличить его от прямого требования.
6. Перед реализацией кратко назвать пользователю:
   - какие specs прочитаны;
   - что именно подтверждено;
   - есть ли часть задачи вне specs.
7. Если требование не найдено, сразу остановиться и сказать: "Такого требования нет в `spec/`." Реализацию не начинать без отдельного подтверждения пользователя.
8. Если пользователь подтверждает новое требование, сначала обновить соответствующие specs, и только после этого переходить к реализации кода.
9. Если агент считает полезным добавить что-то сверх `spec/`, сначала предупредить пользователя, явно назвать, что именно выходит за границы specs, и получить согласование. Самостоятельно добавлять это в specs или код нельзя.
10. Если требование найдено частично, реализовывать только подтвержденную часть. Неподтвержденную часть вынести отдельно и запросить подтверждение.

## Язык

- Все проектные тексты, комментарии, пояснения, specs и UI-тексты должны быть на русском языке.
- Технические имена, ключи данных, маршруты, команды и синтаксис кода можно оставлять на английском, если этого требует стек или существующая структура.

## Как искать требования

- Для общего понимания проекта читать `spec/global-spec.md` и `spec/functional-map.md`.
- Для пользовательского поведения читать `spec/user-stories/`.
- Для конкретной функции читать `spec/feature-specs/`.
- Для архитектуры, данных, маршрутов, валидации и локального хранения читать `spec/technical-specs/`.
- Для порядка изменения требований читать `spec/technical-specs/change-management.md`.

## Запреты

- Не придумывать новые функции, поля данных, сценарии, страницы, статусы, интеграции, платежи, админку, серверную часть или медицинские консультации, если этого нет в specs первой версии.
- Не переносить будущие возможности из разделов "будущая онлайн-версия" в текущую реализацию без отдельного подтверждения.
- Не расширять требования сверх `spec/` по собственной инициативе без согласования с пользователем.
- Не менять продуктовые specs, чтобы самостоятельно легализовать новую функцию или новое поведение.
- Не считать эту карту спецификаций источником требований. При конфликте всегда побеждает актуальный файл в `spec/`.

## Разрешенные самостоятельные действия

- Исправлять орфографию, пунктуацию, грамматику и форматирование без изменения смысла.
- Делать bugfix, который возвращает код к уже описанному в specs поведению.
- Выбирать техническую реализацию, структуру компонентов, helper-функции, repository/service слой и UI-детали, если они не добавляют новое продуктовое поведение.
- Делать технические выводы из specs, если они нужны для выполнения описанного требования и не добавляют новые функции, поля, маршруты, интеграции или бизнес-правила.

## Когда нужно остановиться и спросить

- Требования нет в specs.
- Specs противоречат друг другу.
- Изменение добавляет новое поведение, новые данные, маршруты, интеграции, оплату, авторизацию, аналитику, админку или серверный API.
- Меняется медицинский смысл, условия покупки, статус продаж, доступ в Telegram или ожидания пользователя.
- Есть сомнение, входит ли изменение в specs.

## Детальный рабочий протокол

Подробные правила для новых требований, конфликтов specs, bugfix, будущих идей, черновиков specs и оценки specs находятся в `spec/technical-specs/change-management.md`.

## Проверка перед реализацией

Для обычных и рискованных задач использовать короткий формат:

```text
Проверка specs:
- Прочитано: ...
- Подтверждено: ...
- Вне specs: ...
- Решение: ...
```

Для мелких языковых и форматных исправлений без изменения смысла полный блок не нужен.

## Проверка после изменений

- Проверка должна соответствовать риску изменения.
- Для изменений кода, UI, форм, маршрутов, хранения, валидации и бизнес-логики запускать подходящие проверки, если они доступны.
- Если проверку выполнить нельзя, сообщить об этом пользователю.

## Git

- Не выполнять git-действия без прямой просьбы пользователя.
- По умолчанию дать пользователю команды, список файлов и пример сообщения commit для ручного выполнения git.

## Быстрый маршрут

Если задача неочевидна, начни с `references/spec-map.md`, затем открой перечисленные там specs.
