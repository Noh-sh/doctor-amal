# User Stories проекта Doctor Amal

## Назначение

Эта папка описывает пользовательские истории текущей online-версии Doctor Amal.

User stories нужны не для замены `global-spec.md`, `functional-map.md`, feature specs и technical specs, а для проверки:

- понятен ли путь пользователя на одной Taplink-странице;
- нет ли лишних шагов между кнопками, блоками и внешними платформами;
- не пропущены ли важные состояния без подтвержденных ссылок или данных;
- не создает ли интерфейс ожидание формы, онлайн-оплаты, медицинской консультации или автоматического доступа;
- можно ли проверить сценарий вручную после реализации.

## Источники

Истории основаны на:

- `spec/global-spec.md`;
- `spec/functional-map.md`;
- `spec/feature-specs/README.md`;
- `spec/feature-specs/doctor-block.md`;
- `spec/feature-specs/courses-block.md`;
- `spec/feature-specs/course-purchase-link.md`;
- `spec/feature-specs/external-links.md`;
- `spec/feature-specs/medical-content-rules.md`;
- `spec/feature-specs/future-online-purchase.md`.

## Состав

- `personas.md` — основные типы пользователей и их контекст.
- `core-user-stories.md` — главные истории пользователя текущей Taplink-версии.
- `edge-case-stories.md` — исключительные ситуации текущей Taplink-версии.
- `journey-checklist.md` — проверочный список пользовательских путей.
- `admin-user-stories.md` — текущие истории доктора как редактора контента через админку; не относятся к публичному сценарию Taplink-страницы.

## Как использовать user stories

Перед реализацией функции AI должен:

1. Найти соответствующий общий spec или feature spec.
2. Проверить связанные user stories.
3. Проверить, что сценарий не выходит за границы текущей версии.
4. Реализовать только подтвержденное поведение.
5. Пройти acceptance criteria из user story и `journey-checklist.md`.

## Формат истории

Каждая история описывается так:

```text
Как [тип пользователя],
я хочу [цель],
чтобы [польза].
```

Дополнительно указываются:

- контекст пользователя;
- путь пользователя;
- ожидаемый результат;
- acceptance criteria;
- что не должно происходить.

## Важное правило

Истории не должны добавлять новые продуктовые обещания.

Если user story конфликтует с `global-spec.md`, `functional-map.md` или feature specs, нужно сначала согласовать и обновить specs, а не менять реализацию напрямую.

User stories не должны описывать отдельный каталог курсов, публичные внутренние страницы, статьи, формы заявок, поиск, фильтры, онлайн-оплату, auth покупателей, личный кабинет или сбор персональных данных.

Истории админки описывают текущую защищенную админку. Новые admin-функции не должны добавляться без обновления feature и technical specs.
