# User Stories проекта Doctor-Amal

## Назначение
Эта папка описывает реальные пользовательские истории для проекта Doctor-Amal.

User stories нужны не для замены feature specs и technical specs, а для проверки:
- понятен ли путь пользователя;
- нет ли лишних шагов;
- не пропущены ли важные состояния;
- правильно ли работают ограничения продаж;
- не обещает ли интерфейс медицинскую помощь, оплату или автоматический Telegram-доступ;
- можно ли проверить сценарий вручную после реализации.

## Источники
Истории основаны на:
- `spec/global-spec.md`;
- `spec/functional-map.md`;
- `spec/feature-specs/`;
- `spec/technical-specs/`.

## Состав
- `personas.md` - основные типы пользователей и их контекст.
- `core-user-stories.md` - главные истории пользователя.
- `edge-case-stories.md` - истории с ошибками, закрытыми продажами и пустыми состояниями.
- `journey-checklist.md` - проверочный список пользовательских путей.

## Как использовать user stories
Перед реализацией функции AI должен:
1. Найти соответствующую feature spec.
2. Найти соответствующую technical spec.
3. Проверить связанные user stories.
4. Реализовать сценарий.
5. Пройти acceptance criteria из user story.

## Формат истории
Каждая история описывается так:

```text
Как [тип пользователя],
я хочу [цель],
чтобы [польза].
```

Дополнительно указываются:
- контекст пользователя;
- путь пользователя;
- ожидаемый результат;
- acceptance criteria;
- что не должно происходить.

## Важное правило
Истории не должны добавлять новые продуктовые обещания.

Если user story конфликтует с feature specs или technical specs, нужно сначала обновить specs, а не менять реализацию напрямую.
