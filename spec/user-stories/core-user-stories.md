# User Stories: Основные сценарии

## US-01: Первое знакомство с проектом

### История
Как пользователь, который впервые открыл Doctor-Amal, я хочу понять, кто ведет проект и что здесь можно получить, чтобы решить, стоит ли изучать курсы и материалы дальше.

### Контекст
Пользователь не знает доктора и не понимает формат курсов.

### Путь пользователя
1. Открывает главную страницу.
2. Читает краткое описание проекта.
3. Видит основные темы здоровья.
4. Переходит на страницу "О докторе".
5. Читает подход и роль курсов.
6. Переходит к каталогу или статьям.

### Ожидаемый результат
Пользователь понимает:
- курсы являются образовательными материалами;
- сайт не ставит диагноз;
- оплата не выполняется на сайте;
- доступ к курсу предоставляется через закрытый Telegram-канал после согласованной оплаты.

### Acceptance criteria
- Главная страница дает переход к "О докторе", курсам и статьям.
- Страница "О докторе" объясняет подход без обещаний результата.
- Пользователь не видит кнопку заявки, если продажи закрыты.
- Пользователь видит заявку на консультацию только при открытых продажах.

### Не должно происходить
- Нельзя обещать лечение или гарантированный результат.
- Нельзя предлагать онлайн-оплату.
- Нельзя создавать ощущение медицинской консультации внутри сайта.

## US-02: Поиск курса по теме

### История
Как пользователь, который интересуется конкретной темой здоровья, я хочу найти курс по теме или симптому, чтобы понять, есть ли подходящие материалы.

### Контекст
Пользователь ищет материалы по коже, ЖКТ, гормонам, воспалению, дефицитам или образу жизни.

### Путь пользователя
1. Открывает каталог курсов.
2. Вводит запрос в поиск.
3. Использует фильтр по теме.
4. Смотрит карточки курсов.
5. Открывает страницу подходящего курса.

### Ожидаемый результат
Пользователь видит список опубликованных курсов, статус продаж и понятный переход к подробной странице.

### Acceptance criteria
- Поиск работает по названию, теме, категории и симптомам.
- Фильтры не показывают неопубликованные курсы.
- Симптомы используются только для поиска, не для диагностики.
- Карточка курса показывает статус продаж.

### Не должно происходить
- Нельзя показывать черновики.
- Нельзя выводить диагноз на основе запроса.
- Нельзя скрывать статус продаж.

## US-03: Изучение страницы курса при открытых продажах

### История
Как пользователь, который нашел интересный курс, я хочу открыть страницу курса и понять условия участия, чтобы решить, оставить ли заявку.

### Контекст
Продажи курса открыты.

### Путь пользователя
1. Открывает каталог.
2. Выбирает курс.
3. Читает полное описание.
4. Видит стоимость или пояснение о стоимости.
5. Видит, что продажи открыты.
6. Читает условия покупки и Telegram-доступа.
7. Переходит к заявке на покупку или заявке на консультацию по подбору курса.

### Ожидаемый результат
Пользователь понимает формат курса и может оставить заявку.

### Acceptance criteria
- Кнопка "Оставить заявку на покупку" доступна только при `salesStatus = open`.
- Заявка открывается с выбранным курсом.
- Страница объясняет, что оплата не выполняется на сайте.
- Страница объясняет, что Telegram-доступ предоставляется вручную.

### Не должно происходить
- Нельзя показывать оплату картой.
- Нельзя обещать автоматическое добавление в Telegram.
- Нельзя раскрывать закрытые материалы курса.

## US-04: Заявка на покупку курса

### История
Как пользователь, который выбрал курс, я хочу оставить заявку на покупку, чтобы доктор или ассистент мог связаться со мной по дальнейшим шагам.

### Контекст
Продажи курса открыты.

### Путь пользователя
1. Переходит со страницы курса на форму заявки.
2. Видит выбранный курс.
3. Вводит имя.
4. Вводит контакт для связи.
5. Подтверждает, что заявка не является медицинской консультацией.
6. Подтверждает, что оплата не выполняется на сайте.
7. Отправляет заявку.
8. Видит сообщение об успехе.

### Ожидаемый результат
Заявка сохраняется локально, пользователь получает понятное подтверждение.

### Acceptance criteria
- Без имени заявку отправить нельзя.
- Без контакта заявку отправить нельзя.
- Без обязательных согласий заявку отправить нельзя.
- При закрытых продажах отправка невозможна.
- После успешной отправки показывается правильное сообщение.

### Не должно происходить
- Нельзя просить платежные данные.
- Нельзя очищать форму при ошибке сохранения.
- Нельзя показывать ложный успех, если заявка не сохранилась.

## US-05: Заявка на консультацию по подбору курса

### История
Как пользователь, который сомневается в выборе курса, я хочу оставить заявку на консультацию по подбору курса, чтобы получить помощь в выборе подходящих материалов.

### Контекст
Продажи открыты. Пользователь не уверен, какой курс выбрать.

### Путь пользователя
1. Открывает страницу "О докторе", каталог или страницу курса.
2. Переходит к заявке на консультацию по подбору курса.
3. Вводит имя и контакт.
4. Указывает возраст.
5. Описывает основную проблему.
6. Выбирает темы здоровья.
7. Пишет вопросы, если нужно.
8. Подтверждает обязательные согласия.
9. Отправляет заявку.

### Ожидаемый результат
Заявка сохраняется локально, пользователь понимает, что это помощь в подборе курса, а не медицинская консультация внутри сайта.

### Acceptance criteria
- Форма доступна только при открытых продажах.
- Возраст обязателен и должен быть от 12 до 120.
- Описание проблемы должно быть достаточно подробным.
- Контакт должен быть email, Telegram username или телефоном.
- Успешное сообщение не обещает медицинскую консультацию.

### Не должно происходить
- Нельзя предлагать форму при закрытых продажах.
- Нельзя принимать пустое или слишком короткое описание проблемы.
- Нельзя просить анализы, документы или диагнозы.

## US-06: Чтение статей

### История
Как пользователь, который хочет сначала познакомиться с подходом доктора, я хочу читать бесплатные статьи, чтобы лучше понять темы проекта.

### Контекст
Пользователь может быть на любом этапе: до выбора курса, при закрытых продажах или после просмотра курса.

### Путь пользователя
1. Открывает раздел статей.
2. Видит список опубликованных материалов.
3. Выбирает статью.
4. Читает полный текст.
5. Возвращается к списку, каталогу или странице "О докторе".

### Ожидаемый результат
Пользователь получает ознакомительный материал без давления на покупку.

### Acceptance criteria
- Показываются только опубликованные статьи.
- Статья без полного текста не открывается как полноценный материал.
- Страница статьи не ведет напрямую к заявке на покупку.
- При закрытых продажах статьи остаются доступными.

### Не должно происходить
- Нельзя выдавать статьи за персональные рекомендации.
- Нельзя обещать результат.
- Нельзя раскрывать закрытые материалы курса.

## US-07: Пользователь пришел при закрытых продажах

### История
Как пользователь, который заинтересовался курсом при закрытых продажах, я хочу понять, почему заявка недоступна и что можно делать сейчас.

### Контекст
Продажи курса или всех курсов закрыты.

### Путь пользователя
1. Открывает каталог или страницу курса.
2. Видит статус "Продажи закрыты".
3. Не видит кнопки заявки.
4. Читает объяснение.
5. Переходит к статьям или Telegram-материалам.

### Ожидаемый результат
Пользователь понимает, что сайт работает в ознакомительном режиме.

### Acceptance criteria
- Заявка на покупку недоступна.
- Заявка на консультацию по подбору курса недоступна.
- Пользователь видит предложение читать статьи.
- Есть текст, что перед следующим открытием продаж будет объявление.

### Не должно происходить
- Нельзя показывать форму заявки.
- Нельзя оставлять активный CTA, который ведет пользователя к отправке заявки.
- Нельзя создавать ощущение ошибки сайта.
