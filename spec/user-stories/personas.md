# User Stories: Персоны

## Цель
Описать основные типы пользователей, для которых проверяются сценарии Doctor-Amal.

Персоны не являются медицинскими профилями и не используются для диагностики. Они нужны только для проверки удобства интерфейса и полноты пользовательских путей.

## Persona 1: Пользователь впервые знакомится с проектом

### Контекст
Пользователь впервые открыл сайт Doctor-Amal. Он не знает доктора, не понимает формат курсов и хочет спокойно разобраться, чему можно доверять.

### Цели
- понять, кто ведет проект;
- понять подход доктора;
- понять, что такое курсы;
- узнать, где находятся материалы;
- не столкнуться с агрессивной продажей.

### Риски
- пользователь может принять сайт за медицинскую консультацию;
- пользователь может ожидать онлайн-оплату;
- пользователь может не понять, почему курс проходит через Telegram.

## Persona 2: Пользователь ищет курс по своей проблеме

### Контекст
Пользователь уже знает, что у него есть интерес к теме здоровья, питания, кожи, ЖКТ, гормонов, воспаления или дефицитов. Он хочет найти подходящий курс.

### Цели
- найти курс по теме или симптому;
- понять, подходит ли курс;
- увидеть статус продаж;
- понять, что делать дальше.

### Риски
- пользователь может ожидать диагностику по симптомам;
- пользователь может не найти курс;
- пользователь может попытаться оставить заявку при закрытых продажах.

## Persona 3: Пользователь сомневается в выборе курса

### Контекст
Пользователь видит несколько курсов или не уверен, какой курс ему подходит. Продажи открыты.

### Цели
- оставить заявку на консультацию по подбору курса;
- описать проблему;
- получить подтверждение, что заявка сохранена;
- понять, что это не медицинская консультация внутри сайта.

### Риски
- пользователь может описать проблему слишком коротко;
- пользователь может не оставить контакт;
- пользователь может ожидать срочную медицинскую помощь.

## Persona 4: Пользователь хочет купить выбранный курс

### Контекст
Пользователь выбрал курс, продажи открыты, он хочет оставить заявку.

### Цели
- перейти со страницы курса к заявке;
- увидеть выбранный курс;
- оставить имя и контакт;
- понять, что оплата будет обсуждаться вне сайта;
- получить подтверждение.

### Риски
- пользователь может ожидать оплату картой на сайте;
- пользователь может ожидать автоматический доступ в Telegram;
- пользователь может отправить заявку несколько раз.

## Persona 5: Пользователь пришел, когда продажи закрыты

### Контекст
Пользователь интересуется курсом, но продажи сейчас закрыты.

### Цели
- понять, почему нельзя оставить заявку;
- прочитать статьи;
- изучить подход доктора;
- узнать, что перед следующим открытием продаж будет объявление.

### Риски
- пользователь может искать кнопку покупки;
- пользователь может думать, что сайт сломан;
- пользователь может попытаться попасть на форму заявки прямой ссылкой.

## Persona 6: Владелец проекта или доктор в будущем

### Контекст
В первой версии владелец проекта не имеет админки. В будущем он может управлять курсами, статьями, заявками и периодами продаж.

### Цели
- понимать, какие данные понадобятся в будущем;
- не ломать текущую структуру;
- подготовить проект к онлайн-версии.

### Риски
- преждевременно добавить админку;
- начать хранить лишние персональные данные;
- добавить оплату или Telegram-автоматизацию без отдельной спецификации.
