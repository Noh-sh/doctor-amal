# Feature specs проекта Doctor Amal

## Роль папки

Эта папка описывает поведение отдельных функций текущей online-версии Doctor Amal и отдельно обозначенных будущих расширений.

Актуальная версия проекта описана в:

- `spec/global-spec.md`;
- `spec/functional-map.md`.

Feature specs должны раскрывать эти требования по отдельным блокам и действиям пользователя. Они не должны добавлять функции, которых нет в `global-spec.md` и `functional-map.md`.

## Текущая концепция

Публичная часть — одна online Taplink-страница `/`. Контент хранится в Supabase. Доктор редактирует разрешенный контент через защищенный маршрут `/admin`.

В текущую версию входят:

- фото и имя доктора;
- блок `О докторе`;
- блок `Курсы`;
- кнопки `О докторе`, `Курсы`, `Telegram`, `WhatsApp`, `YouTube`, `Instagram`;
- кнопка `Купить` в блоке курса с переходом в Telegram к менеджеру;
- короткое медицинское ограничение;
- внешние переходы на подтвержденные платформы.

В текущую версию не входят:

- отдельные внутренние страницы;
- отдельный внутренний каталог курсов с маршрутами;
- формы заявок;
- сбор и хранение данных пользователя;
- онлайн-оплата;
- личный кабинет;
- авторизация покупателей;
- управление Auth users через админку;
- физическое удаление курсов из Supabase;
- автоматическая диагностика;
- медицинские статьи, схемы лечения и персональные рекомендации.

## Актуальные feature specs

Feature specs ведутся по такой структуре:

1. `doctor-block.md` — блок `О докторе`.
2. `courses-block.md` — блок `Курсы`.
3. `course-purchase-link.md` — кнопка `Купить` и переход в Telegram к менеджеру.
4. `external-links.md` — внешние кнопки и поведение ссылок.
5. `medical-content-rules.md` — медицинские ограничения пользовательских текстов.
6. `future-online-purchase.md` — будущая онлайн-покупка, не входящая в текущую версию.
7. `admin-content-editing.md` — текущая админка для редактирования контента доктором.
8. `admin-photo-management.md` — загрузка и замена фото доктора через админку.
9. `admin-course-removal.md` — безопасное удаление курсов через админку без физического удаления row.

`admin-content-editing.md` описывает текущую админку. Любое расширение админки сверх этого spec требует связанных user stories, technical specs и отдельного work plan.

## Правила написания feature specs

Каждый feature spec должен содержать:

- назначение функции;
- где пользователь видит функцию;
- что пользователь видит;
- какие действия доступны пользователю;
- основное поведение;
- ограничения текущей версии;
- исключительные ситуации;
- обязательные и необязательные данные;
- что не входит в текущую версию;
- ссылку на отдельный future-spec, если функция не входит в текущую версию.

Feature specs не должны:

- добавлять новые маршруты без подтверждения владельца проекта;
- добавлять формы и сбор данных в текущую версию;
- добавлять онлайн-оплату в текущую версию;
- обещать лечение, выздоровление или медицинский результат;
- превращать проект в медицинскую консультацию;
- копировать технические specs вместо описания пользовательского поведения.

Общие ограничения текущей версии не нужно механически повторять в каждом feature spec полным списком. Достаточно оставить ограничения, которые прямо относятся к конкретной функции, и ссылаться на общую рамку из `global-spec.md`, `functional-map.md` и `medical-content-rules.md`.

## Приоритет чтения

Перед работой над feature specs нужно читать:

1. `spec/global-spec.md`;
2. `spec/functional-map.md`;
3. актуальный feature spec из этой папки;
4. связанные technical specs, когда начнется реализация кода.

Устаревший или future-spec нельзя использовать как основание для изменения текущей версии без обновления актуальных specs.
