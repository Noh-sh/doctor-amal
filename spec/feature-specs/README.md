# Feature specs проекта Doctor Amal

## Роль папки

Эта папка описывает поведение отдельных функций первой локальной версии Doctor Amal.

Актуальная первая версия проекта описана в:

- `spec/global-spec.md`;
- `spec/functional-map.md`.

Feature specs должны раскрывать эти требования по отдельным блокам и действиям пользователя. Они не должны добавлять функции, которых нет в `global-spec.md` и `functional-map.md`.

## Текущая концепция первой версии

Первая версия — локальная одностраничная web-страница в формате Taplink.

В первую версию входят:

- фото и имя доктора;
- блок `О докторе`;
- блок `Курсы`;
- кнопки `О докторе`, `Курсы`, `Telegram`, `WhatsApp`, `YouTube`, `Instagram`;
- кнопка `Купить` в блоке курса с переходом в Telegram к менеджеру;
- короткое медицинское ограничение;
- внешние переходы на подтвержденные платформы.

В первую версию не входят:

- отдельные внутренние страницы;
- отдельный внутренний каталог курсов с маршрутами;
- формы заявок;
- сбор и хранение данных пользователя;
- онлайн-оплата;
- личный кабинет;
- админка;
- серверная часть;
- автоматическая диагностика;
- медицинские статьи, схемы лечения и персональные рекомендации.

## Актуальные feature specs новой Taplink-версии

Новые feature specs нужно вести по такой структуре:

1. `doctor-block.md` — блок `О докторе`.
2. `courses-block.md` — блок `Курсы`.
3. `course-purchase-link.md` — кнопка `Купить` и переход в Telegram к менеджеру.
4. `external-links.md` — внешние кнопки и поведение ссылок.
5. `medical-content-rules.md` — медицинские ограничения пользовательских текстов.
6. `future-online-purchase.md` — будущая онлайн-покупка, не входящая в первую локальную версию.
7. `admin-content-editing.md` — будущая админка для редактирования контента доктором, не входящая в первую локальную версию.

На текущем этапе созданы:

- `doctor-block.md`;
- `courses-block.md`;
- `course-purchase-link.md`;
- `external-links.md`;
- `medical-content-rules.md`;
- `future-online-purchase.md`.
- `admin-content-editing.md`.

Все feature specs из текущего списка созданы.

`admin-content-editing.md` относится к будущей онлайн-версии с админкой. Его нельзя использовать как основание для немедленной реализации без связанных user stories, technical specs и отдельного work plan.

## Правила написания feature specs

Каждый feature spec должен содержать:

- назначение функции;
- где пользователь видит функцию;
- что пользователь видит;
- какие действия доступны пользователю;
- основное поведение;
- ограничения первой локальной версии;
- исключительные ситуации;
- что нужно уточнить перед реализацией, если данных пока нет;
- что не входит в первую локальную версию;
- связь с будущей онлайн-версией, если она есть.

Feature specs не должны:

- добавлять новые маршруты без подтверждения владельца проекта;
- добавлять формы и сбор данных в первую локальную версию;
- добавлять онлайн-оплату в первую локальную версию;
- обещать лечение, выздоровление или медицинский результат;
- превращать проект в медицинскую консультацию;
- копировать технические specs вместо описания пользовательского поведения.

Общие ограничения первой локальной версии не нужно механически повторять в каждом feature spec полным списком. Достаточно оставить ограничения, которые прямо относятся к конкретной функции, и ссылаться на общую рамку из `global-spec.md`, `functional-map.md` и `medical-content-rules.md`.

## Приоритет чтения

Перед работой над feature specs нужно читать:

1. `spec/global-spec.md`;
2. `spec/functional-map.md`;
3. актуальный feature spec из этой папки;
4. связанные technical specs, когда начнется реализация кода.

Если feature specs еще не переписаны под новую Taplink-версию, нельзя использовать старый файл как основание для реализации новой функции.
