# Feature Spec: Безопасное удаление курсов через админку

## Назначение

Этот spec описывает новую подтвержденную возможность админки: доктор может убрать курс из админки и публичной страницы через безопасное удаление.

В текущем этапе безопасное удаление означает soft delete или архивирование, а не физическое удаление записи из Supabase.

## Где доктор видит функцию

Функция доступна только в защищенной админке:

```text
/admin
```

Обычный посетитель публичной страницы не видит удаленные курсы и не может влиять на список курсов.

## Кто может удалить курс

Удалить курс может только активный `doctor_admin`.

Факт входа в Supabase Auth без активной роли `doctor_admin` не дает право удалять или архивировать курс.

## Что доктор может сделать

Доктор может:

- выбрать существующий курс в админке;
- нажать действие удаления;
- увидеть предупреждение, что курс исчезнет из публичной страницы и основного списка админки;
- подтвердить удаление;
- получить понятный feedback после успешного удаления или ошибки.

После успешного удаления:

- курс не показывается на публичной странице;
- курс не оставляет пустой блок или пустое место;
- курс не показывается в основном списке редактируемых курсов в админке;
- публичная страница `/` обновляется сразу или почти сразу через текущий revalidation-механизм.

## Технический смысл удаления

В этом этапе удаление курса реализуется как безопасное архивирование:

- запись курса остается в Supabase;
- запись получает технический признак удаления, например `deleted_at`;
- публичная страница читает только активные неудаленные курсы;
- админка показывает в основном списке только неудаленные курсы.

Физический SQL `delete` для курсов в этом этапе не используется.

## Почему не физическое удаление

Физическое удаление повышает риск случайной потери данных и требует отдельного процесса восстановления.

Текущий подтвержденный вариант дает доктору ожидаемое поведение “курс удален из интерфейса”, но сохраняет возможность технического восстановления через Supabase при ошибке.

## Что не входит

В этот этап не входят:

- физическое удаление row из Supabase через UI;
- корзина удаленных курсов;
- список архива в админке;
- восстановление курса через UI;
- массовое удаление;
- audit log;
- подтверждение через email или отдельную роль.

Если позже потребуется корзина или восстановление через UI, нужны отдельные specs.

## Ошибки

Если удаление не удалось, админка должна показать понятную ошибку без stack trace, SQL/PostgREST details, policy details, UUID Auth user или других служебных данных.

Если revalidation не сработал после успешного удаления, админка показывает понятный warning/error: курс удален из Supabase, но публичная страница может обновиться по обычному `revalidate: 30`.

## Связь с публичной страницей

Удаление курса не добавляет:

- отдельные публичные маршруты;
- новый каталог;
- формы заявок;
- оплату;
- сбор данных пользователя.

Публичный блок `Курсы` остается accordion на одной странице `/`.
