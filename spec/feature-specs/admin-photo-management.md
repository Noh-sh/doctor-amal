# Feature Spec: Фото доктора через админку

## Назначение

Этот spec описывает новую подтвержденную возможность админки: доктор может загрузить или заменить фото, которое показывается на публичной Taplink-странице.

Функция нужна, чтобы доктор мог обновлять визуальный первый блок страницы без обращения к разработчику.

## Где доктор видит функцию

Функция доступна только в защищенной админке:

```text
/admin
```

Обычный посетитель публичной страницы не видит загрузку фото и не может отправлять файлы.

## Кто может менять фото

Фото может загружать или заменять только активный `doctor_admin`.

Факт входа в Supabase Auth без активной роли `doctor_admin` не дает право загружать файл или менять `doctor_profile.photo_src`.

## Что доктор может сделать

Доктор может:

- выбрать файл изображения;
- увидеть понятное состояние выбранного файла;
- сохранить новое фото;
- заменить текущее фото новым;
- получить понятный feedback после успешного сохранения или ошибки.

После успешной загрузки и сохранения:

- `doctor_profile.photo_src` обновляется на путь или публичный URL нового файла;
- `doctor_profile.photo_alt` получает безопасный русскоязычный alt-текст;
- публичная страница `/` обновляется сразу или почти сразу через текущий revalidation-механизм.

## Допустимые файлы

Допустимые форматы:

- JPEG;
- PNG;
- WebP.

Максимальный размер файла: `5 MB`.

Если файл больше лимита или имеет неподдерживаемый тип, админка должна показать понятную ошибку и не сохранять изменение.

## Alt-текст

Фото должно иметь пользовательский русскоязычный `alt`.

Минимально допустимый вариант для текущей реализации:

```text
Фото: {имя доктора}
```

Отдельное ручное редактирование alt-текста не является обязательным для первого этапа загрузки фото.

## Хранение

Файл хранится в Supabase Storage.

Storage bucket, пути, policies и права доступа описываются в technical specs и migration.

Frontend не должен использовать service role key для загрузки фото.

## Замена фото

При замене фото новое изображение должно загрузиться и сохраниться в `doctor_profile` только после успешной проверки файла.

Если загрузка файла прошла, но обновление `doctor_profile` не удалось, админка должна показать понятную ошибку. Публичная страница не должна перейти в состояние с битой ссылкой на фото.

Удаление старого storage-файла может выполняться только после успешного сохранения нового фото и только если это безопасно для текущих данных. Ошибка удаления старого файла не должна ломать публичную страницу.

## Что не входит

В этот этап не входят:

- загрузка нескольких фото;
- галерея;
- обрезка фото внутри приложения;
- фильтры, ретушь или обработка изображения;
- загрузка медицинских документов;
- загрузка файлов публичными пользователями;
- управление Storage через service role во frontend;
- хранение персональных медицинских данных.

## Ошибки

Админка должна показывать понятные русские сообщения без stack trace, bucket policy details, signed URL details, access token, email, UUID Auth user или других служебных данных.

Если Supabase Storage временно недоступен, фото не считается обновленным.

Если публичная страница не смогла мгновенно обновиться после успешной загрузки, админка использует тот же feedback, что и другие сохранения: данные сохранены, но публичная страница может обновиться по обычному `revalidate: 30`.

## Связь с публичной страницей

Публичная страница остается одной Taplink-страницей `/`.

Загрузка фото не добавляет:

- новые публичные маршруты;
- формы для посетителя;
- сбор данных пользователя;
- оплату;
- запись на консультацию;
- медицинские обещания.
