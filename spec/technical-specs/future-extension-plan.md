# Technical Spec: План будущего расширения

## Цель
Описать, как проект Doctor-Amal должен расширяться после первой локальной версии на Next.js.

Этот документ нужен, чтобы новые функции можно было добавлять без полной переделки архитектуры, данных и пользовательских сценариев.

## Базовая точка первой версии
Первая версия:
- Next.js + TypeScript;
- App Router;
- локальные JSON-файлы для курсов и статей;
- `localStorage` для пользовательских заявок;
- без онлайн-оплаты;
- без личного кабинета;
- без автоматического добавления в Telegram;
- без медицинской диагностики;
- без серверной базы данных.

## Главный принцип расширения
Новые функции должны добавляться через существующие слои:
- UI pages и components;
- domain models;
- repository/service layer;
- validation layer;
- storage/API layer.

Нельзя добавлять новую функцию так, чтобы страница напрямую зависела от конкретного способа хранения данных. Например, страница каталога не должна напрямую читать базу данных, JSON или внешний API. Она должна получать данные через repository/service.

## Что нельзя ломать при расширении
При любом будущем изменении нельзя нарушать следующие правила:
- сайт не принимает оплату картой напрямую;
- проект не хранит платежные данные;
- курс остается образовательным продуктом;
- заявка на подбор курса не становится медицинской консультацией внутри сайта;
- сайт не ставит диагноз;
- сайт не назначает лечение;
- доступ в закрытый Telegram-канал предоставляется вручную после согласованной оплаты;
- пользователь не видит черновики, служебные поля и внутренние идентификаторы;
- основные пользовательские тексты остаются на русском языке.

## Этап 1: Локальная версия
Цель этапа - реализовать все основные пользовательские сценарии без сервера.

Должно быть реализовано:
- главная страница;
- страница "О докторе";
- каталог курсов;
- страница курса;
- список статей;
- страница статьи;
- форма заявки на покупку курса;
- форма заявки на консультацию по подбору курса;
- локальное сохранение заявок;
- обработка ошибок и пустых состояний.

Данные:
- курсы: JSON;
- статьи: JSON;
- заявки: `localStorage`.

## Этап 2: Server API для заявок
Цель этапа - заменить локальное сохранение заявок на серверную обработку.

Что можно добавить:
- `app/api/requests/route.ts`;
- серверную валидацию заявки;
- запись заявок в базу данных;
- защиту от повторной отправки на сервере;
- техническое логирование ошибок;
- уведомление доктору или ассистенту о новой заявке.

Что должно остаться прежним:
- UI формы;
- типы заявок `course_purchase` и `pre_purchase_consultation`;
- пользовательские сообщения об успехе и ошибке;
- запрет онлайн-оплаты;
- запрет платежных данных.

Технический контракт:
```ts
interface RequestRepository {
  listRequests(): Promise<UserRequest[]>;
  saveRequest(request: UserRequest): Promise<SaveRequestResult>;
}
```

В локальной версии этот repository использует `localStorage`. В онлайн-версии он может отправлять `POST /api/requests`.

## Этап 3: База данных
Цель этапа - перенести курсы, статьи и заявки из локального хранения в базу данных.

Что можно добавить:
- таблицу или коллекцию `courses`;
- таблицу или коллекцию `articles`;
- таблицу или коллекцию `requests`;
- миграции схемы;
- seed-скрипты для начальных данных.

Модели должны соответствовать `data-model.md`.

Минимальные сущности:
- `Course`;
- `Article`;
- `UserRequest`.

Что нельзя добавлять в базу:
- платежные данные;
- данные банковских карт;
- пароли пользователей, если личного кабинета еще нет;
- медицинские документы;
- результаты анализов.

## Этап 4: Административная панель
Цель этапа - дать владельцу проекта или доктору возможность управлять контентом и заявками.

Что можно добавить:
- `/admin`;
- авторизацию администратора;
- просмотр заявок;
- изменение статуса заявки;
- управление курсами;
- управление статьями;
- управление статусами продаж;
- управление периодами открытия продаж.

Админка не должна быть доступна обычному пользователю.

Возможные статусы заявки:
- `new`;
- `needs_clarification`;
- `course_recommended`;
- `awaiting_offline_payment`;
- `telegram_access_granted`;
- `closed`;
- `cancelled`.

Важно: статус `telegram_access_granted` означает только ручную отметку администратора. Сайт не должен автоматически добавлять пользователя в Telegram-канал.

## Этап 5: Управление продажами
Цель этапа - управлять периодами открытия продаж без правки кода.

Что можно добавить:
- отдельную сущность `salesPeriod`;
- глобальный статус продаж;
- статус продаж для отдельного курса;
- дату начала и конца продаж;
- текст объявления о будущем открытии продаж;
- автоматическое скрытие кнопок заявки после завершения периода.

Правило приоритета:
- если курс имеет собственный статус продаж, он важнее глобального статуса;
- если собственного статуса нет, используется глобальный статус;
- если период завершился, заявки на покупку должны быть недоступны.

## Этап 6: Уведомления
Цель этапа - сообщать доктору или ассистенту о новых заявках.

Что можно добавить:
- email-уведомление;
- Telegram bot notification для администратора;
- internal notification в админке.

Уведомление не должно содержать лишние чувствительные данные. Минимально достаточно:
- тип заявки;
- имя;
- контакт;
- выбранный курс, если есть;
- дата заявки.

## Этап 7: Экспорт данных
Цель этапа - дать владельцу проекта возможность выгрузить заявки.

Что можно добавить:
- экспорт CSV;
- экспорт JSON;
- фильтр по дате;
- фильтр по типу заявки;
- фильтр по статусу.

Экспорт должен быть доступен только администратору.

Экспорт не должен содержать платежные данные, потому что проект их не собирает.

## Этап 8: Онлайн-контент и SEO
Цель этапа - улучшить публичную онлайн-версию.

Что можно добавить:
- metadata для страниц;
- sitemap;
- robots.txt;
- Open Graph изображения;
- человекочитаемые URL;
- публичные страницы статей;
- публичные страницы курсов.

SEO не должно менять медицинские ограничения проекта. Заголовки и описания не должны обещать лечение, диагностику или гарантированный результат.

## Возможные будущие API routes
Если проект переходит к серверной версии, можно добавить:

```text
GET    /api/courses
GET    /api/courses/:courseId
GET    /api/articles
GET    /api/articles/:articleId
POST   /api/requests
GET    /api/admin/requests
PATCH  /api/admin/requests/:requestId
POST   /api/admin/courses
PATCH  /api/admin/courses/:courseId
POST   /api/admin/articles
PATCH  /api/admin/articles/:articleId
```

Публичные API не должны отдавать:
- черновики;
- внутренние названия Telegram-каналов;
- служебные заметки администратора;
- технические поля, не предназначенные пользователю.

## Миграционный путь от локальной версии к онлайн-версии
Рекомендуемый порядок:
1. Оставить UI без изменений.
2. Добавить server-side `RequestRepository`.
3. Заменить сохранение заявок из `localStorage` на `POST /api/requests`.
4. Подключить базу данных для заявок.
5. Перенести курсы и статьи из JSON в базу.
6. Добавить админку.
7. Добавить управление продажами.
8. Добавить уведомления.
9. Добавить экспорт.

Такой порядок снижает риск, потому что сначала меняется хранение заявок, а не весь пользовательский интерфейс.

## Контроль совместимости
Перед добавлением новой функции нужно проверить:
- не меняет ли она смысл существующих feature specs;
- не требует ли она сбора новых персональных данных;
- не нарушает ли она запрет онлайн-оплаты;
- не превращает ли заявку в медицинскую консультацию;
- не требует ли обновления `data-model.md`;
- не требует ли обновления `requests-and-validation.md`;
- не требует ли обновления `routing-and-ui.md`.

## Когда нужно обновлять specs
Specs нужно обновить до реализации, если добавляется:
- новая страница;
- новая сущность данных;
- новый тип заявки;
- новый статус курса или заявки;
- новый способ хранения данных;
- интеграция с внешним сервисом;
- административная функция;
- изменение правил продаж.

## Итоговое правило
Любое будущее расширение должно сохранять текущую пользовательскую логику и границы ответственности проекта.

Технически проект должен развиваться от локальной Next.js версии к серверной Next.js версии через замену repository/storage layer, а не через переписывание всех страниц.
