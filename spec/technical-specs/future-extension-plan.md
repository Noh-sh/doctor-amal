# Technical Spec: План будущего расширения

## Цель

Описать возможные будущие расширения текущей online-версии Doctor Amal без смешивания будущих функций с актуальными требованиями.

Этот документ не является разрешением на реализацию будущих функций. Любой следующий этап требует отдельного подтверждения владельца проекта и обновления specs.

## Текущая базовая точка

Текущая версия:

- Next.js + TypeScript и App Router;
- публичная Taplink-страница `/`;
- защищенная админка `/admin`;
- Supabase как основной источник контента;
- локальный fallback публичного контента;
- Supabase Auth только для доктора;
- роль `doctor_admin` через таблицу `admin_users`;
- редактирование профильных текстов, внешних ссылок, курсов, цен и Telegram-ссылки менеджера;
- загрузка и замена фото доктора через админку;
- создание, скрытие и безопасное удаление курсов без физического удаления;
- кнопка `Купить` ведет в Telegram к менеджеру;
- нет auth покупателей, заявок, сбора данных публичного пользователя и онлайн-оплаты.

## Главный принцип расширения

Будущие функции должны добавляться как новые подтвержденные этапы, а не незаметно проникать в текущую версию.

Перед любым расширением нужно:

1. подтвердить требование с владельцем проекта;
2. обновить `global-spec.md` или `functional-map.md`, если меняется граница продукта;
3. добавить или обновить feature specs;
4. обновить user stories;
5. обновить technical specs;
6. только потом менять код.

## Будущие функции, не реализовывать без новых specs

- физическое удаление курсов из Supabase и восстановление после физического удаления;
- корзина удаленных курсов и восстановление через UI;
- управление Auth users и назначением ролей через интерфейс;
- роли менеджера, помощника или второго администратора;
- черновики, предпросмотр и отдельная публикация;
- audit log;
- аналитика и UTM-метки;
- отдельные публичные страницы;
- поиск и фильтры курсов;
- формы, заявки, auth покупателей и личный кабинет;
- онлайн-покупка и автоматическая выдача доступа.

Каждая функция из списка требует отдельного feature spec, technical spec, user stories, work plan и проверки безопасности.

## Возможная онлайн-покупка

Онлайн-покупка не входит автоматически в версию с CMS/админкой.

Если позже она понадобится, нужно отдельно описать:

- какие курсы можно покупать внутри проекта;
- какие данные пользователя собираются;
- какой платежный провайдер используется;
- как обрабатывается успешная оплата;
- как обрабатывается ошибка оплаты;
- как пользователь получает доступ к курсу или закрытому Telegram-каналу;
- какие юридические тексты, согласия, условия оплаты и возврата нужны;
- где хранятся данные покупки;
- нужна ли админка для просмотра покупок.

## Безопасность будущих функций

При любом будущем расширении:

- редактировать данные может только авторизованный пользователь;
- обычный пользователь не видит админские маршруты;
- нельзя хранить лишние медицинские данные;
- нельзя хранить платежные данные без отдельной платежной архитектуры и юридического основания;
- ошибки админки не должны ломать публичную страницу;
- внешние URL должны проверяться перед публикацией.

## Порядок будущего расширения

1. Подтвердить новое продуктовое требование.
2. Обновить актуальные specs и future-план.
3. Создать новую migration, если меняются schema, grants или RLS.
4. Применить migration через Supabase CLI.
5. Реализовать код.
6. Проверить RLS через CLI/MCP.
7. Проверить публичную страницу и `/admin`.

## Контроль совместимости

Будущее расширение не должно ломать:

- одну понятную Taplink-структуру публичной страницы;
- медицинские ограничения;
- отсутствие обещаний результата;
- внешний Telegram-сценарий покупки, пока он остается подтвержденным;
- русскоязычный интерфейс;
- запрет на неподтвержденные URL, цены и медицинские формулировки.
