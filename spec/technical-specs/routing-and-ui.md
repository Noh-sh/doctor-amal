# Technical Spec: Маршруты и UI

## Цель
Зафиксировать страницы, маршруты, навигацию, компоненты и состояния интерфейса первой версии Doctor-Amal.

## Маршруты
Рекомендуемые маршруты:

```text
/                         Главная страница
/doctor                   О докторе
/courses                  Каталог курсов
/courses/:courseId        Страница курса
/request                  Форма заявки
/articles                 Список статей
/articles/:articleId      Страница статьи
/*                        Страница 404
```

Форма заявки должна поддерживать query params:

```text
/request?type=course_purchase&courseId=skin-health-basics
/request?type=pre_purchase_consultation
```

Если query params некорректны, форма должна показать безопасное состояние и попросить выбрать доступный тип заявки.

## Глобальная навигация
В основной навигации должны быть ссылки:
- Главная;
- О докторе;
- Курсы;
- Статьи.

Кнопка заявки показывается только когда есть открытые продажи. Если все продажи закрыты или скоро откроются, глобальная навигация не должна предлагать заявку.

## Главная страница
Главная страница должна:
- кратко объяснить проект Doctor-Amal;
- показать основные темы: кожа, ЖКТ, гормональный баланс, головные боли, хроническое воспаление, дефициты, психологическое состояние;
- дать переход в каталог курсов;
- дать переход на страницу "О докторе";
- дать переход к статьям;
- показать заявку на консультацию по подбору курса только если продажи открыты.

Главная страница не должна быть маркетинговой страницей с обещаниями результата. Тон должен быть спокойным, образовательным и медицински аккуратным.

## Страница "О докторе"
UI должен содержать блоки:
- публичная информация о докторе;
- подход и ценности;
- роль курсов;
- формат закрытых Telegram-каналов;
- почему нет онлайн-оплаты;
- зачем нужна консультация по подбору курса;
- действия: каталог, статьи, заявка при открытых продажах.

Если данные страницы неполные, незаполненные блоки скрываются.

## Каталог курсов
Компоненты:
- поисковая строка;
- фильтр по теме или категории;
- фильтр по статусу продаж;
- список карточек курсов;
- пустое состояние;
- ошибка загрузки.

Карточка курса показывает:
- название;
- короткое описание;
- категорию;
- темы или симптомы;
- стоимость или текст "Стоимость уточняется";
- статус продаж;
- период продаж, если известен;
- кнопку "Подробнее".

Кнопка "Купить" в каталоге необязательна. Если она используется, она показывается только при `salesStatus = "open"`.

## Страница курса
Страница курса показывает:
- название;
- полное описание;
- категорию;
- темы и симптомы;
- пользу от материалов курса;
- стоимость;
- статус продаж;
- условия покупки;
- объяснение доступа через закрытый Telegram-канал;
- блок похожих курсов, если есть опубликованные связанные курсы.

Действия зависят от статуса:
- `open`: показать "Оставить заявку на покупку" и "Нужна помощь с выбором";
- `closed`: не показывать заявки, предложить статьи;
- `coming_soon`: не показывать заявки, показать период и предложить статьи.

## Форма заявки
Форма должна иметь два режима:
- покупка выбранного курса;
- консультация по подбору курса.

Общие поля:
- имя;
- контакт для связи;
- согласие, что заявка не является медицинской консультацией;
- согласие, что оплата не выполняется на сайте.

Поля покупки:
- выбранный курс;
- комментарий.

Поля консультации:
- возраст;
- основные симптомы или проблема;
- выбранные темы здоровья;
- вопросы пользователя;
- желаемый курс или предпочтение.

Кнопка отправки должна быть заблокирована во время сохранения, чтобы избежать двойной отправки.

## Статьи
Список статей показывает:
- заголовок;
- краткое описание;
- тему;
- дату публикации;
- источник, если указан.

Страница статьи показывает:
- заголовок;
- дату;
- тему;
- полный текст;
- источник;
- переход назад к списку статей;
- обычную навигацию к каталогу и странице "О докторе".

Страница статьи не должна вести сразу к заявке на покупку.

## Состояния интерфейса
Каждая страница, которая зависит от данных, должна поддерживать:
- loading;
- success;
- empty;
- not found;
- error.

Даже если данные читаются синхронно из JSON, UI должен быть спроектирован так, чтобы позже можно было заменить чтение на API-запрос.

## Правила показа действий
Нужно использовать единую функцию или сервис:

```ts
function canSubmitCoursePurchase(course: Course): boolean {
  return course.publicationStatus === "published" && course.salesStatus === "open";
}

function canSubmitPrePurchaseConsultation(hasAnyOpenSales: boolean): boolean {
  return hasAnyOpenSales;
}
```

Эти правила должны применяться одинаково на главной, в каталоге, на странице курса и в форме заявки.

## UI-тексты статусов
Рекомендуемые тексты:
- `open`: "Продажи открыты";
- `closed`: "Продажи закрыты";
- `coming_soon`: "Продажи скоро откроются";
- нет цены: "Стоимость уточняется";
- нет периода: не показывать период или показать "Период продаж будет объявлен отдельно".

## Ошибки форм
Ошибки должны показываться рядом с полями и в общем summary над формой, если ошибок несколько.

Форма не должна очищать введенные данные после ошибки сохранения.

После успешного сохранения форма может:
- показать экран успеха;
- предложить вернуться в каталог;
- предложить читать статьи.

## 404 и недоступные сущности
Если курс или статья не найдены:
- показать понятное сообщение;
- дать переход в ближайший раздел;
- не показывать технический stack trace.

Если курс или статья существуют, но не опубликованы:
- обычный пользователь видит состояние "недоступно";
- черновой контент не показывается.

## Адаптивность
Интерфейс должен работать на:
- мобильной ширине от 360px;
- планшете;
- десктопе.

Формы на мобильном должны быть одноколоночными. Карточки курсов могут переходить из сетки в список.

## Доступность
Обязательные требования:
- у каждого input есть label;
- ошибки связаны с полями через доступный текст;
- интерактивные элементы доступны с клавиатуры;
- фокус не теряется после отправки формы;
- цвет статуса дублируется текстом.
