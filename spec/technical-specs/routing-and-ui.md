# Technical Spec: Маршруты и UI

## Цель

Зафиксировать маршрут, структуру страницы, UI-компоненты и состояния интерфейса первой локальной Taplink-версии Doctor Amal.

## Маршруты первой версии

В первой локальной версии публичный пользовательский маршрут один:

```text
/    Taplink-страница Doctor Amal
```

Допустим служебный маршрут:

```text
/*   not-found для неизвестных внутренних URL
```

В первой версии не должно быть пользовательских маршрутов:

```text
/doctor
/courses
/courses/:courseId
/request
/articles
/articles/:articleId
/admin
```

Если пользователь попадает на неизвестный внутренний URL, приложение показывает понятную 404-страницу. 404-страница не должна раскрывать старую структуру проекта и должна давать возможность вернуться на главную страницу `/`.

## Структура страницы

Рекомендуемый порядок:

1. Первый блок:
   - фото доктора;
   - имя доктора;
   - краткое представление, если оно подтверждено;
   - короткое медицинское ограничение `Информация на странице носит ознакомительный характер.`
2. Кнопки первой версии:
   - `О докторе`;
   - `Курсы`;
   - `Telegram`;
   - `WhatsApp`;
   - `YouTube`;
   - `Instagram`.
3. Блок `О докторе`.
4. Блок `Курсы`.

Финальное расположение блоков может быть уточнено при UI-реализации, если оно не меняет сценарии и не добавляет новое поведение.

## Поведение кнопок

### `О докторе`

Кнопка ведет к блоку `О докторе` на той же странице.

Допустимые реализации:

- scroll к секции;
- раскрытие секции;
- другое одностраничное поведение без отдельного маршрута.

### `Курсы`

Кнопка ведет к блоку `Курсы` на той же странице.

### Внешние кнопки

Кнопки `Telegram`, `WhatsApp`, `YouTube`, `Instagram`:

- видны в интерфейсе первой версии;
- активны только при подтвержденном URL;
- без URL остаются видимыми, но неактивными;
- не должны выглядеть как рабочая ссылка без подтвержденного URL.

Правило открытия внешних ссылок должно быть подтверждено перед реализацией: текущая вкладка, новая вкладка или стандартное поведение браузера.

## Блок "О докторе"

Блок может показывать только подтвержденные данные:

- имя доктора;
- фото, если используется повторно;
- краткое представление;
- образование или места обучения;
- знания, опыт и профессиональные направления;
- направления здоровья и недомогания, которые можно упоминать;
- подходы и форматы помощи.

Если фото доктора еще не предоставлено, вместо фото можно показывать согласованный временный placeholder. Placeholder не должен использовать реальную фотографию другого человека и не должен создавать впечатление, что это фото доктора.

Блок не должен:

- открывать отдельный маршрут;
- содержать форму связи;
- предлагать запись на консультацию;
- собирать данные пользователя;
- звучать как медицинская консультация;
- обещать лечение или результат.

## Блок "Курсы"

В свернутом виде курса показываются:

- название курса;
- цена курса или подтвержденная текстовая формулировка цены.

После нажатия на название курса показываются:

- описание курса;
- кнопка `Купить`.

Цена не дублируется в раскрытом курсе, если specs не изменены.

Курс раскрывается через accordion на этой же странице.

Accordion должен быть удобен на мобильной ширине: название курса, цена, описание и кнопка `Купить` не должны переполнять экран, перекрывать друг друга или требовать горизонтального скролла.

Accordion не должен создавать отдельный маршрут, корзину, форму заявки, онлайн-оплату или сбор данных.

## Кнопка "Купить"

Кнопка `Купить` видна только после раскрытия конкретного курса.

Если Telegram-ссылка менеджера подтверждена:

- кнопка является внешней ссылкой в Telegram;
- используется одна общая Telegram-ссылка для всех курсов;
- название выбранного курса не передается автоматически в первой версии;
- пользователь самостоятельно сообщает менеджеру, какой курс его интересует.

Если Telegram-ссылка не подтверждена:

- кнопка видима только как неактивная;
- она не должна выглядеть как рабочий внешний переход;
- текст неактивной кнопки: `Покупка временно недоступна`.

Кнопка `Купить` не должна:

- создавать заказ;
- открывать форму покупки;
- запускать оплату;
- сохранять данные пользователя;
- показывать сообщение об успешной покупке;
- обещать автоматический доступ в закрытый Telegram-канал.

## UI-состояния

Нужны состояния:

- подтвержденные данные показаны;
- URL внешней платформы не подтвержден: внешняя кнопка показывает `Ссылка будет добавлена позже`;
- Telegram-ссылка менеджера не подтверждена: кнопка `Купить` показывает `Покупка временно недоступна`;
- список курсов не подтвержден;
- цена курса не подтверждена;
- локальные данные проекта повреждены или неполные.

Пустые и ошибочные состояния должны быть спокойными и пользовательскими. Нельзя показывать stack trace или служебные поля.

## Медицинское ограничение

На странице должен быть короткий текст:

```text
Информация на странице носит ознакомительный характер.
```

Другие медицинские ограничения работают как внутренние правила проверки текстов: не обещать выздоровление, не ставить диагноз, не давать персональные рекомендации и не раскрывать схемы лечения как инструкции.

## Адаптивность

Интерфейс должен работать на:

- мобильной ширине от 360px;
- планшете;
- десктопе.

Taplink-страница должна быть удобной на мобильном экране: кнопки достаточно крупные, текст не переполняет контейнеры, блоки читаются без горизонтального скролла.

## Доступность

Обязательные требования:

- интерактивные элементы доступны с клавиатуры;
- accordion имеет доступные состояния;
- активные и неактивные кнопки различимы не только цветом;
- внешние ссылки имеют понятный текст;
- изображения имеют корректный `alt`, если несут смысл.
