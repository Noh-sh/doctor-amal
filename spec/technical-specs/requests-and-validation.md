# Technical Spec: Заявки и валидация

## Цель

Зафиксировать, что в текущей Taplink-версии нет публичных заявок и форм, а также описать валидацию контента публичной страницы и админки.

## Публичные заявки

В текущей версии не поддерживаются заявки:

- `course_purchase`;
- `pre_purchase_consultation`;
- любые другие формы обращения.

Пользователь не вводит данные на странице Doctor Amal. Покупка курса начинается через внешний переход в Telegram к менеджеру, если ссылка подтверждена.

## Публичные формы

На публичной странице нет:

- формы покупки;
- формы заявки;
- формы консультации;
- поля имени;
- поля телефона;
- поля email;
- поля Telegram username;
- медицинской анкеты;
- загрузки файлов публичным пользователем;
- согласий формы.

Если в будущем потребуется форма, это будет новым требованием: сначала обновляются specs, потом реализация.

## Валидация контента страницы

Перед использованием локальных данных проекта или данных из Supabase нужно проверить:

- есть ли имя доктора;
- есть ли корректный путь к фото, если фото используется;
- если фото загружается через админку, файл имеет допустимый тип и размер;
- все внешние URL либо `null`, либо валидные URL;
- кнопка с `url = null` не активна;
- у курса есть название;
- описание курса не пустое, если курс показывается как раскрываемый;
- цена задана подтвержденным текстом;
- Telegram-ссылка менеджера либо `null`, либо валидный Telegram URL;
- текст медицинского ограничения равен подтвержденному тексту или согласованной замене.

Админка содержит только формы редактирования контента. Они не являются публичными пользовательскими формами и не создают заявки.

## Валидация админки

- имя доктора обязательно;
- краткое представление и профильные списки необязательны;
- название, описание и цена курса обязательны;
- URL внешней ссылки обязателен только при `is_enabled = true`;
- Telegram-ссылка менеджера необязательна;
- если Telegram-ссылка менеджера отсутствует, текст неактивной кнопки обязателен.
- фото через админку принимает только JPEG, PNG или WebP до `5 MB`;
- удаление курса через админку выполняется как soft delete после подтверждения.

## Проверка внешних URL

Минимальная проверка URL:

- строка начинается с `https://` или другого подтвержденного формата внешней платформы;
- URL соответствует ожидаемой платформе, если это можно проверить без сложной интеграции;
- пустой URL хранится как `null`, а не как пустая строка.

Нельзя самостоятельно исправлять или придумывать URL.

## Проверка Telegram-ссылки менеджера

Перед активной кнопкой `Купить` нужно проверить:

- `managerTelegramUrl` задан;
- ссылка подтверждена владельцем проекта.

В текущей версии техническая реализация не должна придумывать формат сообщения, prefilled message, URL-параметры или Telegram bot для автоматической передачи названия курса. Пользователь самостоятельно сообщает менеджеру, какой курс его интересует.

## Ошибки контента

Ошибки локального контента или данных Supabase не являются ошибками пользователя.

UI должен:

- показывать неактивное состояние вместо рабочей ссылки без URL;
- не показывать неподтвержденный курс как реальный;
- не показывать неподтвержденную цену как сумму;
- не показывать stack trace;
- не собирать данные пользователя как обходной путь.

## Недопустимые действия

Система не должна:

- принимать заявку;
- сохранять обращение;
- валидировать пользовательский контакт;
- просить платежные данные;
- создавать заказ;
- показывать успешную покупку;
- обещать автоматический доступ в Telegram;
- превращать кнопку `Купить` в оплату.
