# Technical Spec: Заявки и валидация

## Цель
Описать технические правила создания, проверки и сохранения заявок.

## Типы заявок
Поддерживаются два типа:

```text
course_purchase
pre_purchase_consultation
```

Другие типы в первой версии не создаются.

## Инициализация формы

### Покупка курса
URL:

```text
/request?type=course_purchase&courseId=<courseId>
```

Система должна:
- найти курс по `courseId`;
- проверить, что курс опубликован;
- проверить, что продажи открыты;
- предзаполнить выбранный курс;
- если курс не найден или продажи закрыты, заблокировать отправку и показать объяснение.

### Консультация по подбору курса
URL:

```text
/request?type=pre_purchase_consultation
```

Система должна:
- проверить, есть ли хотя бы один курс с открытыми продажами;
- если открытых продаж нет, не принимать заявку и предложить статьи.

## Валидация общих полей

### name
Правила:
- обязательное поле;
- trim перед проверкой;
- минимум 2 символа;
- максимум 80 символов.

Ошибка:

```text
Укажите имя, чтобы мы могли обработать заявку.
```

### contact
Правила:
- обязательное поле;
- trim перед проверкой;
- минимум 3 символа;
- максимум 120 символов;
- допускаются email, Telegram username или номер телефона.

Минимальная техническая проверка:
- email должен содержать `@` и доменную часть;
- Telegram username может начинаться с `@` и должен иметь минимум 5 символов после `@`;
- телефон может содержать цифры, пробелы, `+`, `-`, скобки.

Если контакт не похож на email, Telegram username или телефон, система должна попросить пользователя проверить поле. Это ограничение снижает риск сохранить контакт, по которому невозможно ответить на заявку.

Ошибка пустого поля:

```text
Укажите контакт для связи, чтобы доктор или ассистент мог ответить на заявку.
```

Ошибка некорректного поля:

```text
Проверьте контакт для связи. Возможно, в нем есть ошибка.
```

### consentNotMedicalConsultation
Обязательно `true`.

Текст согласия:

```text
Я понимаю, что заявка не является медицинской консультацией и не заменяет очный медицинский осмотр.
```

### consentNoOnlinePayment
Обязательно `true`.

Текст согласия:

```text
Я понимаю, что оплата не выполняется на сайте и обсуждается отдельно с доктором или ассистентом.
```

## Валидация заявки на покупку

### courseId
Правила:
- обязательное поле;
- курс должен существовать;
- курс должен быть опубликован;
- `salesStatus` должен быть `open`.

Если курс не выбран:

```text
Выберите курс, по которому хотите оставить заявку.
```

Если продажи закрыты:

```text
Продажи курса сейчас закрыты. Заявки на покупку не принимаются.
```

### comment
Правила:
- необязательное поле;
- trim перед сохранением;
- максимум 1000 символов.

Если поле длиннее лимита, система просит сократить комментарий.

## Валидация заявки на консультацию по подбору курса

### age
Правила:
- обязательное поле для заявки на консультацию по подбору курса;
- должно быть числом от 12 до 120;
- если пользователь младше 18 лет, UI должен показывать аккуратное предупреждение о необходимости участия взрослого при медицинских вопросах.

### mainProblem
Правила:
- обязательное поле;
- trim перед проверкой;
- минимум 20 символов;
- максимум 2000 символов.

Ошибка:

```text
Опишите вопрос подробнее, чтобы доктор или ассистент мог понять, какой курс может быть уместен.
```

### selectedTopics
Правила:
- массив строк;
- значения должны быть из известного списка тем;
- минимум 1 тема желательно, но не обязательно, если `mainProblem` достаточно подробен.

### questions
Правила:
- необязательное поле;
- максимум 1500 символов.

### preferredCourseId
Правила:
- необязательное поле;
- если заполнено, курс должен существовать и быть опубликован.

## Создание заявки
После успешной валидации система создает объект заявки.

Обязательные технические поля:
- `id`;
- `type`;
- `status = "new"`;
- `createdAt`;
- `updatedAt`;
- пользовательские поля по типу заявки.

`id` можно создавать через timestamp + случайный suffix:

```text
req_<yyyymmddhhmmss>_<random>
```

В будущем сервер может заменить этот формат.

## Защита от повторной отправки
Перед сохранением нужно проверить похожие заявки, созданные недавно.

Похожая заявка:
- тот же `type`;
- тот же нормализованный `contact`;
- тот же `courseId` для покупки;
- создана за последние 10 минут.

Если похожая заявка найдена, система должна предупредить пользователя:

```text
Похожая заявка уже была отправлена недавно. Проверьте данные перед повторной отправкой.
```

В первой версии допустимо не сохранять повторную заявку без явного повторного подтверждения. Более простой вариант: заблокировать повторную отправку и оставить форму заполненной.

## Сохранение
Сервис сохранения должен возвращать:

```ts
type SaveRequestResult =
  | { ok: true; requestId: string }
  | { ok: false; errorCode: "storage_unavailable" | "validation_failed" | "duplicate_recent" | "unknown" };
```

UI не должен показывать пользователю `errorCode` напрямую.

## Успешные сообщения

Для покупки:

```text
Заявка на покупку курса отправлена. Доктор или ассистент свяжется с вами для обсуждения оплаты и доступа к закрытому Telegram-каналу.
```

Для консультации:

```text
Заявка отправлена. Доктор или ассистент сможет уточнить детали и помочь подобрать подходящий курс.
```

## Ошибка сохранения
Если сохранение не удалось:
- не показывать сообщение об успехе;
- оставить данные в форме;
- показать понятную ошибку;
- предложить повторить попытку.

Текст:

```text
Заявку не удалось сохранить. Проверьте данные и попробуйте еще раз.
```

## Недопустимые действия
Система не должна:
- принимать заявку на покупку при закрытых продажах;
- принимать заявку без контакта;
- принимать заявку без обязательных согласий;
- просить платежные данные;
- предлагать оплату внутри сайта;
- обещать автоматический доступ в Telegram;
- создавать заявку как медицинскую консультацию.
