# Technical Specs проекта Doctor-Amal

## Назначение
Эта папка описывает техническую часть проекта Doctor-Amal на основе документов из `spec/` и `spec/feature-specs/`.

Цель technical specs - дать разработчику или AI достаточно технического контекста, чтобы реализовывать локальную версию без дополнительных вопросов по архитектуре, данным, маршрутам, состояниям интерфейса, валидации и ограничениям проекта.

## Источники требований
Технические спецификации основаны на:
- `spec/global-spec.md`;
- `spec/functional-map.md`;
- `spec/feature-specs/doctor-block.md`;
- `spec/feature-specs/courses-block.md`;
- `spec/feature-specs/course-purchase-link.md`;
- `spec/feature-specs/external-links.md`;
- `spec/feature-specs/medical-content-rules.md`;
- `spec/feature-specs/future-online-purchase.md`.

Часть technical specs может еще относиться к старой структуре проекта с каталогом, заявками, статьями и локальным хранением заявок. Перед реализацией новой Taplink-версии technical specs нужно последовательно переписать под актуальные `global-spec.md`, `functional-map.md` и feature specs новой версии.

Если технический документ и feature spec конфликтуют, сначала нужно проверить, не менялось ли продуктовое требование. Без отдельного решения владельца проекта техническая реализация не должна менять смысл feature spec.

## Состав technical specs
- `architecture.md` - платформа, стек, слои приложения, ограничения первой версии.
- `data-model.md` - сущности, поля, типы, статусы, связи и примеры JSON.
- `routing-and-ui.md` - страницы, маршруты, навигация, компоненты и состояния интерфейса.
- `requests-and-validation.md` - техническое поведение форм, валидация, ошибки, защита от повторной отправки.
- `local-storage.md` - структура локальных данных, чтение, запись, миграции, обработка поврежденных файлов.
- `future-extension-plan.md` - путь расширения от локальной версии к онлайн-версии, API, базе, админке и уведомлениям.
- `change-management.md` - безопасный порядок изменений после завершения проекта, связанные specs и regression checklist.
- `implementation-checklist.md` - критерии готовности, тестовые сценарии и правила приемки.

## Общие технические решения для первой версии
- Первая версия реализуется как локально запускаемое web-приложение на Next.js.
- Первая локальная версия является одной страницей в формате Taplink.
- Приложение не должно требовать сервера, базы данных, авторизации, личного кабинета, онлайн-оплаты, админки или сбора данных пользователя.
- Данные для отображения страницы могут храниться локально, если это не добавляет новый пользовательский сценарий и не противоречит актуальным specs.
- Онлайн-оплата, платежные данные, личный кабинет, медицинская диагностика, формы заявок и автоматический доступ в Telegram-канал не реализуются.
- Все пользовательские тексты по умолчанию на русском языке.
- Внутренние технические идентификаторы не показываются пользователю.
- Все медицинские формулировки должны оставаться образовательными и ознакомительными.

## Обязательный стек для реализации
Первая версия проекта должна разрабатываться на:
- Framework: Next.js.
- Language: TypeScript.
- Routing: Next.js App Router.
- UI: React components внутри Next.js.
- Styling: CSS Modules, глобальный CSS или другой легкий подход, совместимый с Next.js.
- Runtime data: локальные JSON-файлы для seed-данных курсов и статей.
- User requests: не используются в первой Taplink-версии без отдельного подтверждения владельца проекта и обновления specs.

Next.js выбран как основной стек, потому что проект планируется развивать в онлайн-версию. Первая локальная версия должна использовать такую структуру, чтобы позже можно было добавить серверное API, базу данных и административную часть без переписывания пользовательских сценариев.

## Обязательные ограничения
Приложение не должно:
- принимать оплату на сайте;
- запрашивать или хранить данные банковских карт;
- автоматически добавлять пользователя в закрытый Telegram-канал;
- ставить диагноз;
- назначать лечение;
- обещать медицинский результат;
- хранить медицинские документы, анализы, пароли или лишние персональные данные;
- показывать черновики и неопубликованные курсы/статьи обычному пользователю.

## Термины
- Курс - образовательный продукт, доступ к которому после согласованной оплаты предоставляется через закрытый Telegram-канал.
- Кнопка `Купить` - внешняя ссылка в Telegram для связи с менеджером, а не платежная кнопка.
- Внешняя кнопка - кнопка перехода на подтвержденную внешнюю платформу.
- Неактивная кнопка - видимая кнопка без рабочего перехода, если точная ссылка еще не подтверждена.
