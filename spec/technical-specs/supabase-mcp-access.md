# Technical Spec: Supabase MCP access

## Цель

Описать правила текущего read-only MCP-доступа Codex к Supabase project Doctor Amal.

MCP подключен для инспекции и проверки. Этот spec запрещает использовать его как обход migrations и актуальных specs.

## Статус этапа

MCP-доступ допускается только после того, как:

- Supabase content-source specs подтверждены;
- структура базы описана в `supabase/migrations/`;
- migration применен к online Supabase project через Supabase CLI;
- RLS включен на таблицах контента;
- public policies разрешают только чтение опубликованного или активного контента.

MCP является проверочным слоем после `spec -> CLI migrations`.

## Назначение MCP

Разрешенное назначение MCP:

- инспекция текущей структуры Supabase project;
- проверка наличия таблиц, колонок, constraints и RLS policies;
- проверка, что migration применен к remote project;
- чтение технической информации, нужной для сверки specs, migrations и фактического состояния Supabase;
- диагностика расхождений между `supabase/migrations/` и online project.

MCP не является источником правды по требованиям. Источник правды остается в `spec/`.

MCP не заменяет `supabase/migrations/`. Источник правды по структуре базы остается в migration-файлах.

## Граница доступа

Разрешен только inspect/read workflow.

Разрешено:

- читать список таблиц, колонок, constraints, indexes и policies;
- читать список migrations и статус их применения;
- выполнять безопасные select-проверки технического состояния;
- сверять фактическую структуру с `spec/technical-specs/supabase-content-source.md` и `supabase/migrations/`.

Не разрешено без отдельного подтвержденного spec и work plan:

- создавать, менять или удалять таблицы;
- менять RLS policies;
- менять constraints или indexes;
- менять seed-контент;
- записывать пользовательские данные;
- создавать auth-пользователей;
- добавлять заявки, заказы, платежи, аналитику или медицинские данные;
- использовать MCP как основной способ изменения schema.

## Правило изменений базы

Все изменения структуры базы должны проходить через порядок:

```text
spec -> supabase/migrations/ -> npx supabase db push -> MCP verification
```

Если MCP показывает, что remote project отличается от migration-файлов, нельзя исправлять это прямым изменением через MCP. Нужно:

1. описать расхождение в work plan;
2. определить, что является правильным состоянием по specs;
3. создать или исправить migration;
4. применить migration через Supabase CLI;
5. повторно проверить через MCP или CLI.

## Ключи и секреты

Секреты нельзя коммитить в репозиторий.

Запрещено:

- добавлять MCP-токены, Supabase access token, secret key или service role key в git;
- добавлять секреты в `NEXT_PUBLIC_*`;
- вставлять секреты в specs, work plans, roadmap или комментарии к коду;
- использовать service role key для обычной инспекции, если достаточно менее привилегированного доступа.

Если конкретная MCP-интеграция требует токен или ключ, он хранится только локально в пользовательской конфигурации MCP или в локальном `.env`, который не попадает в git.

Если для будущей задачи понадобится write-доступ или service role key, сначала нужен отдельный spec с причиной, границами, рисками и проверкой.

## Связь с текущими Supabase specs

MCP не меняет публичный сценарий пользователя:

- публичная страница остается `/`;
- Supabase остается источником публичного контента страницы;
- UI-компоненты не знают, пришли данные из Supabase или локального fallback;
- кнопка `Купить` остается внешним переходом в Telegram к менеджеру при подтвержденной ссылке;
- формы, заявки, auth покупателей, заказы, online payment, медицинские данные, аналитика и админка не добавляются.

Если MCP используется для проверки данных, он не должен легализовать неподтвержденный контент. Подтвержденность контента определяется specs и владельцем проекта.

## Проверка перед использованием MCP

Перед использованием MCP нужно проверить:

- текущая ветка и рабочее дерево понятны;
- `supabase/migrations/` содержит актуальную структуру базы;
- `npx supabase migration list` показывает ожидаемое состояние local и remote migrations;
- RLS включен на всех таблицах первого Supabase-этапа;
- публичные policies не дают write-доступ;
- локальная MCP-конфигурация не будет коммититься.

## Проверка после использования MCP

После использования MCP нужно зафиксировать в work plan:

- какие MCP-средства подключены;
- какой Supabase project используется;
- какие операции были выполнены;
- подтверждено ли совпадение specs, migrations и remote project;
- что секреты не попали в git.

Если MCP недоступен, нужно указать причину и не обходить это прямыми изменениями базы вне CLI migrations.
