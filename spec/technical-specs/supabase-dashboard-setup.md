# Technical Spec: Подготовка Supabase Dashboard

## Цель

Дать пошаговую инструкцию для подготовки online Supabase project перед подключением Next.js к Supabase.

Этот документ не подключает Supabase к коду приложения. Он нужен, чтобы создать таблицы, RLS policies и начальные записи в Supabase Dashboard или перенести тот же SQL в Supabase CLI migration.

## Перед началом

Нужно открыть:

- Supabase Dashboard;
- новый или выбранный project;
- раздел SQL Editor.

Локальный Supabase через Docker на этом этапе не используется.

Supabase CLI допускается для применения migration к online project, если структура базы должна храниться в git.

## Что будет создано

SQL создает только таблицы контента:

- `doctor_profile`;
- `external_links`;
- `courses`;
- `purchase_settings`;
- `page_settings`.

SQL не создает:

- заявки;
- пользователей;
- auth покупателей;
- заказы;
- платежи;
- медицинские анкеты;
- аналитику;
- админку приложения.

## SQL для Supabase Dashboard

Вставить этот SQL в Supabase Dashboard -> SQL Editor и выполнить.

Если используется Supabase CLI, этот же SQL должен находиться в `supabase/migrations/`.

```sql
create table if not exists public.doctor_profile (
  id text primary key default 'main',
  display_name text not null,
  photo_src text,
  photo_alt text,
  short_intro text,
  education jsonb not null default '[]'::jsonb,
  experience jsonb not null default '[]'::jsonb,
  professional_directions jsonb not null default '[]'::jsonb,
  health_topics jsonb not null default '[]'::jsonb,
  help_formats jsonb not null default '[]'::jsonb,
  is_published boolean not null default false,
  updated_at timestamptz not null default now(),
  constraint doctor_profile_singleton check (id = 'main'),
  constraint doctor_profile_education_array check (jsonb_typeof(education) = 'array'),
  constraint doctor_profile_experience_array check (jsonb_typeof(experience) = 'array'),
  constraint doctor_profile_professional_directions_array check (jsonb_typeof(professional_directions) = 'array'),
  constraint doctor_profile_health_topics_array check (jsonb_typeof(health_topics) = 'array'),
  constraint doctor_profile_help_formats_array check (jsonb_typeof(help_formats) = 'array')
);

create table if not exists public.external_links (
  id bigint generated by default as identity primary key,
  platform text not null,
  label text not null,
  url text,
  is_enabled boolean not null default false,
  inactive_text text,
  sort_order integer not null default 0,
  updated_at timestamptz not null default now(),
  constraint external_links_platform_check check (platform in ('telegram', 'whatsapp', 'youtube', 'instagram')),
  constraint external_links_label_check check (label in ('Telegram', 'WhatsApp', 'YouTube', 'Instagram')),
  constraint external_links_url_enabled_check check (
    is_enabled = false or (url is not null and length(trim(url)) > 0)
  ),
  constraint external_links_platform_unique unique (platform)
);

create table if not exists public.courses (
  id text primary key,
  slug text not null unique,
  title text not null,
  description text not null,
  price_display_text text not null,
  price_is_confirmed boolean not null default false,
  purchase_label text not null default 'Купить',
  is_active boolean not null default true,
  sort_order integer not null default 0,
  updated_at timestamptz not null default now(),
  constraint courses_purchase_label_check check (purchase_label = 'Купить')
);

create table if not exists public.purchase_settings (
  id text primary key default 'main',
  manager_telegram_url text,
  inactive_text text,
  updated_at timestamptz not null default now(),
  constraint purchase_settings_singleton check (id = 'main')
);

create table if not exists public.page_settings (
  id text primary key default 'main',
  medical_notice text not null,
  is_published boolean not null default false,
  updated_at timestamptz not null default now(),
  constraint page_settings_singleton check (id = 'main')
);

alter table public.doctor_profile enable row level security;
alter table public.external_links enable row level security;
alter table public.courses enable row level security;
alter table public.purchase_settings enable row level security;
alter table public.page_settings enable row level security;

do $$
begin
  if not exists (
    select 1 from pg_policies
    where schemaname = 'public'
      and tablename = 'doctor_profile'
      and policyname = 'Public can read published doctor profile'
  ) then
    create policy "Public can read published doctor profile"
      on public.doctor_profile
      for select
      to anon
      using (is_published = true);
  end if;

  if not exists (
    select 1 from pg_policies
    where schemaname = 'public'
      and tablename = 'external_links'
      and policyname = 'Public can read external links'
  ) then
    create policy "Public can read external links"
      on public.external_links
      for select
      to anon
      using (true);
  end if;

  if not exists (
    select 1 from pg_policies
    where schemaname = 'public'
      and tablename = 'courses'
      and policyname = 'Public can read active courses'
  ) then
    create policy "Public can read active courses"
      on public.courses
      for select
      to anon
      using (is_active = true);
  end if;

  if not exists (
    select 1 from pg_policies
    where schemaname = 'public'
      and tablename = 'purchase_settings'
      and policyname = 'Public can read purchase settings'
  ) then
    create policy "Public can read purchase settings"
      on public.purchase_settings
      for select
      to anon
      using (id = 'main');
  end if;

  if not exists (
    select 1 from pg_policies
    where schemaname = 'public'
      and tablename = 'page_settings'
      and policyname = 'Public can read published page settings'
  ) then
    create policy "Public can read published page settings"
      on public.page_settings
      for select
      to anon
      using (is_published = true);
  end if;
end $$;
```

## Проверочные запросы

После выполнения schema и seed можно выполнить эти запросы в SQL Editor.

```sql
select id, display_name, is_published
from public.doctor_profile;

select platform, label, url, is_enabled, sort_order
from public.external_links
order by sort_order;

select slug, title, price_display_text, price_is_confirmed, is_active, sort_order
from public.courses
order by sort_order;

select id, manager_telegram_url, inactive_text
from public.purchase_settings;

select id, medical_notice, is_published
from public.page_settings;
```

Ожидаемый смысл:

- профиль `main` опубликован;
- внешние ссылки видны, но пока без URL и неактивны;
- два курса активны и имеют цену `15 000 руб.`;
- Telegram-ссылка менеджера пока `null`;
- медицинское ограничение опубликовано.

## Где взять ключи после подготовки

После выполнения SQL нужно открыть Supabase Dashboard:

1. Project Settings.
2. API Keys или Data API.
3. Скопировать Project URL.
4. Скопировать publishable key.

Для локальной разработки позже будут нужны:

```env
NEXT_PUBLIC_SUPABASE_URL=...
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=...
DOCTOR_SUPABASE_URL=...
DOCTOR_SUPABASE_SERVICE_ROLE_KEY=
```

Secret key или service role key на этом этапе не нужен и не должен использоваться кодом приложения.

Если для обучения нужно сохранить service role key локально, использовать только server-side имя без `NEXT_PUBLIC_`:

```env
DOCTOR_SUPABASE_SERVICE_ROLE_KEY=
```

Реальный service role key нельзя отправлять в чат, коммитить в git или использовать во frontend-коде.

## Готовность к следующему плану

Можно переходить к подключению кода, когда:

- project в Supabase создан;
- SQL для таблиц выполнен;
- seed выполнен;
- RLS включен;
- проверочные запросы показывают ожидаемые данные;
- Project URL и публичный ключ готовы;
- `.env.local` еще не коммитится и будет создан только на этапе подключения кода.
