# Technical Spec: Supabase как источник контента

## Цель

Описать первый Supabase-этап проекта Doctor Amal.

Supabase на этом этапе используется только как online-источник контента Taplink-страницы. Он не меняет публичный сценарий пользователя и не добавляет новые пользовательские функции.

## Граница этапа

Входит:

- хранение подтвержденного контента страницы в Supabase;
- чтение опубликованного контента публичной страницей `/`;
- преобразование rows из Supabase в текущую модель `TaplinkPageData`;
- online Supabase Dashboard как основной способ подготовки проекта, таблиц, policies и тестовых записей;
- env-переменные для публичного чтения данных.

Не входит:

- заявки;
- формы;
- пользователи;
- auth покупателей;
- личный кабинет;
- заказы;
- онлайн-оплата;
- платежные статусы;
- автоматическая выдача доступа;
- медицинские анкеты;
- персональные медицинские данные;
- аналитика;
- админка внутри приложения;
- MCP-подключение Codex к Supabase.

MCP может быть рассмотрен позже отдельным планом и отдельными правилами доступа.

## Пользовательский сценарий

Публичный сценарий остается прежним:

- пользователь открывает одну страницу `/`;
- видит фото, имя и краткое представление доктора;
- открывает блок `О докторе`;
- открывает блок `Курсы`;
- видит названия и цены курсов;
- раскрывает курс и видит описание и кнопку `Купить`;
- переходит во внешние платформы только по подтвержденным URL;
- переходит в Telegram к менеджеру по кнопке `Купить`, если ссылка подтверждена.

Supabase не должен создавать ожидание, что внутри проекта доступна покупка, форма заявки, консультация, личный кабинет или оплата.

## Основной data flow

```text
Supabase tables
  -> data access layer
  -> mapper/adapter
  -> TaplinkPageData
  -> существующие UI-компоненты Taplink-страницы
```

UI-компоненты не должны знать, пришли данные из локального файла или из Supabase.

Если позже потребуется временный fallback на локальные данные, он должен использовать только уже подтвержденный контент и не должен показывать пользователю технические ошибки.

## Таблицы первого этапа

Имена таблиц технические и могут быть на английском языке.

### `doctor_profile`

Назначение: один опубликованный профиль доктора для Taplink-страницы.

Поля:

- `id` — технический идентификатор, допустимо значение `main`;
- `display_name` — публичное имя доктора;
- `photo_src` — путь или URL фото;
- `photo_alt` — alt-текст фото;
- `short_intro` — короткое представление;
- `education` — список строк или `jsonb`;
- `experience` — список строк или `jsonb`;
- `professional_directions` — список строк или `jsonb`;
- `health_topics` — список строк или `jsonb`;
- `help_formats` — список строк или `jsonb`;
- `is_published` — опубликован ли профиль;
- `updated_at` — техническая дата обновления.

Правила:

- публичная страница читает только опубликованный профиль;
- неподтвержденные биографические факты нельзя добавлять в таблицу как реальные данные;
- фото должно быть предоставлено или подтверждено владельцем проекта.

### `external_links`

Назначение: внешние кнопки `Telegram`, `WhatsApp`, `YouTube`, `Instagram`.

Поля:

- `id` — технический идентификатор;
- `platform` — `telegram`, `whatsapp`, `youtube` или `instagram`;
- `label` — текст кнопки;
- `url` — подтвержденный URL или `null`;
- `is_enabled` — активна ли кнопка;
- `inactive_text` — текст неактивного состояния;
- `sort_order` — порядок отображения;
- `updated_at` — техническая дата обновления.

Правила:

- `is_enabled` должен быть `false`, если `url` не задан;
- кнопка без URL остается видимой, но неактивной;
- URL нельзя придумывать или автоматически исправлять;
- аналитика и UTM-метки не добавляются без отдельного подтверждения.

### `courses`

Назначение: список курсов на Taplink-странице.

Поля:

- `id` — технический идентификатор;
- `slug` — стабильный технический ключ курса;
- `title` — название курса;
- `description` — описание курса;
- `price_display_text` — цена текстом;
- `price_is_confirmed` — подтверждена ли цена;
- `purchase_label` — текст кнопки, в первой версии `Купить`;
- `is_active` — показывать ли курс;
- `sort_order` — порядок отображения;
- `updated_at` — техническая дата обновления.

Правила:

- курс не получает отдельный публичный маршрут;
- курс не получает отдельную Telegram-ссылку;
- цена показывается текстом, без вычисления валюты или платежной логики;
- описание курса должно соблюдать `medical-content-rules.md`;
- неподтвержденные курсы и цены нельзя показывать как реальные.

### `purchase_settings`

Назначение: общие настройки кнопки `Купить`.

Поля:

- `id` — технический идентификатор, допустимо значение `main`;
- `manager_telegram_url` — общая Telegram-ссылка менеджера или `null`;
- `inactive_text` — текст неактивной покупки;
- `updated_at` — техническая дата обновления.

Правила:

- одна общая Telegram-ссылка используется для всех курсов;
- без подтвержденной ссылки кнопка `Купить` не должна быть рабочим переходом;
- название курса не передается автоматически в Telegram на этом этапе;
- Telegram bot и prefilled message требуют отдельных specs.

### `page_settings`

Назначение: общие тексты страницы, которые относятся к текущей модели `TaplinkPageData`.

Поля:

- `id` — технический идентификатор, допустимо значение `main`;
- `medical_notice` — текст медицинского ограничения;
- `is_published` — опубликованы ли настройки;
- `updated_at` — техническая дата обновления.

Правила:

- текущий подтвержденный текст: `Информация на странице носит ознакомительный характер.`;
- изменение медицинского смысла текста требует проверки по `medical-content-rules.md`;
- техническая ошибка получения текста не должна показывать пользователю stack trace.

## Что нельзя хранить в Supabase на первом этапе

Нельзя создавать или хранить:

- имя, телефон, email или Telegram username пользователя;
- медицинские вопросы пользователя;
- медицинские документы;
- результаты анализов;
- заявки;
- заказы;
- платежные данные;
- статусы покупки;
- историю кликов;
- аналитику;
- учетные записи покупателей;
- доступы к закрытым материалам.

Если одно из этих требований появится позже, сначала нужны отдельные specs.

## Online Supabase Dashboard и CLI migrations

Первый этап готовится сразу в online Supabase Dashboard.

После уточнения процесса допускается Supabase CLI без локального Docker, чтобы сохранить структуру базы в git через migration-файлы.

Локальный Supabase через Docker на этом этапе не используется.

Порядок подготовки:

1. Создать Supabase project.
2. Создать таблицы первого этапа через SQL Editor, Table Editor или Supabase CLI migration.
3. Включить RLS на всех таблицах.
4. Добавить policies только для публичного чтения опубликованного контента.
5. Добавить тестовые записи подтвержденного контента.
6. Проверить, что публичный ключ не дает прав на запись.
7. Получить Project URL и публичный ключ.

Рекомендуемый CLI-путь:

```bash
npx supabase login
npx supabase link --project-ref dagykilvpiacfbwpcluv
npx supabase db push
```

Migration-файлы хранятся в `supabase/migrations/`.

## Env-переменные и ключи

Ключи добавляются после создания online Supabase project и до написания Supabase client в коде.

Для публичного чтения нужны:

```env
DOCTOR_SUPABASE_URL=...
DOCTOR_SUPABASE_PUBLISHABLE_KEY=...
```

Если конкретный Supabase project использует legacy keys, допустим вариант:

```env
DOCTOR_SUPABASE_URL=...
DOCTOR_SUPABASE_ANON_KEY=...
```

Правила:

- предпочтительно использовать `DOCTOR_SUPABASE_PUBLISHABLE_KEY`, если он доступен;
- legacy `DOCTOR_SUPABASE_ANON_KEY` допустим, если dashboard показывает legacy anon key;
- `.env.local` создается локально и не коммитится;
- hosting env-переменные добавляются после локальной проверки подключения;
- secret key или service role key нельзя добавлять в `NEXT_PUBLIC_*`;
- secret/service role key не нужен для первого этапа публичного чтения контента;
- если позже появится server-only задача, secret key описывается отдельным spec и хранится только как server-side env, например `DOCTOR_SUPABASE_SERVICE_ROLE_KEY`;
- текущий код не должен использовать `DOCTOR_SUPABASE_SERVICE_ROLE_KEY`.

## Безопасность и RLS

Для всех таблиц первого этапа:

- RLS должен быть включен;
- публичная policy разрешает только `select`;
- публичная policy должна ограничивать чтение опубликованными или активными rows;
- публичные `insert`, `update`, `delete` запрещены;
- запись выполняется только владельцем проекта через Supabase Dashboard на первом этапе;
- приложение не должно использовать service role key во frontend-коде.

Минимальная логика чтения:

- `doctor_profile`: только `is_published = true`;
- `page_settings`: только `is_published = true`;
- `courses`: только `is_active = true`, сортировка по `sort_order`;
- `external_links`: rows читаются для отображения, но активность определяется `url` и `is_enabled`;
- `purchase_settings`: singleton-настройка для общей Telegram-ссылки.

## Валидация данных

Перед передачей данных в UI нужно проверить:

- профиль доктора содержит имя;
- путь или URL фото задан только подтвержденным значением;
- внешние URL либо `null`, либо валидные URL ожидаемой платформы;
- внешняя кнопка без URL не активна;
- у активного курса есть название;
- у активного курса есть описание;
- цена задана подтвержденным текстом или согласованным состоянием;
- Telegram-ссылка менеджера либо `null`, либо валидный Telegram URL;
- медицинское ограничение соответствует подтвержденному тексту или согласованной замене.

Ошибки данных не должны приводить к белому экрану или stack trace для пользователя.

## Связь с будущей онлайн-оплатой

Возможная online payment остается будущим направлением.

Этот spec не разрешает создавать:

- таблицы `orders`;
- таблицы `payments`;
- checkout;
- платежного провайдера;
- платежные статусы;
- сбор данных покупателя;
- автоматическую выдачу доступа.

Если позже владелец проекта подтвердит online payment, нужно создать отдельные feature specs, user stories и technical specs для платежей, данных, маршрутов, ошибок, безопасности, юридических текстов и проверки.

## Готовность перед кодом

Перед подключением Supabase в коде должны быть готовы:

- этот technical spec;
- online Supabase project;
- таблицы первого этапа;
- RLS policies;
- тестовые записи контента;
- публичный Project URL;
- publishable key или legacy anon key;
- локальный `.env.local`, не попавший в git.

Только после этого можно переходить к следующему плану разработки.

## Проверки после будущей реализации

Минимальные проверки после подключения кода:

```bash
npm run build
git diff --check
```

Ручные проверки:

- `/` открывается;
- неизвестные внутренние URL показывают 404;
- данные профиля, курсы, цены и внешние кнопки приходят из Supabase;
- блок `О докторе` раскрывается;
- блок `Курсы` раскрывается;
- внешние кнопки без URL остаются неактивными;
- `Купить` без Telegram-ссылки остается неактивной;
- нет форм, оплаты, заявок и сбора пользовательских данных;
- secret/service role key не используется во frontend-коде;
- `.env.local` не попал в git.
