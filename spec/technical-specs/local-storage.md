# Technical Spec: Локальные данные и хранение

## Цель

Зафиксировать, какие данные первая локальная Taplink-версия может хранить в проекте и какие виды хранения не используются.

## Главное правило

Первая локальная версия не собирает, не сохраняет и не обрабатывает пользовательские данные.

Поэтому в первой версии не используется browser `localStorage` для заявок, обращений, контактов, медицинских данных или истории действий пользователя.

## Допустимые локальные данные проекта

В проекте могут храниться только данные для отображения страницы:

- профиль доктора;
- фото или путь к фото;
- короткое представление;
- текст блока `О докторе`;
- список курсов;
- описания курсов;
- цены или согласованные состояния без цены;
- внешние URL;
- Telegram-ссылка менеджера;
- текст медицинского ограничения;
- тексты неактивных состояний.

Эти данные редактируются в коде или локальных data-файлах разработчиком первой версии.

## Рекомендуемый формат

Допустимые варианты:

- TypeScript-константа с типом `TaplinkPageData`;
- JSON-файл с дополнительной runtime-проверкой структуры;
- несколько небольших файлов по смыслу: `doctor`, `courses`, `links`.

Для первой версии предпочтительно использовать TypeScript-константы, если это проще и снижает риск невалидной структуры.

## Что запрещено хранить

В первой версии нельзя хранить:

- имя пользователя;
- телефон;
- email;
- Telegram username пользователя;
- медицинские вопросы пользователя;
- медицинские документы;
- результаты анализов;
- заявки;
- заказы;
- платежные данные;
- историю кликов;
- аналитику.

## Поведение при неполных данных

Если данные неполные:

- неподтвержденный URL делает кнопку видимой, но неактивной;
- неподтвержденная Telegram-ссылка менеджера делает `Купить` неактивной;
- неподтвержденная цена не заменяется выдуманной суммой;
- неподтвержденный список курсов не заменяется выдуманными курсами;
- неподтвержденные данные о докторе не выводятся как факты.

## Поврежденные локальные данные проекта

Если локальный data-файл поврежден или не соответствует ожидаемой структуре:

- приложение не должно показывать белый экран;
- пользователь видит спокойное состояние или отсутствие неподтвержденного блока;
- техническая причина может быть видна разработчику в development, но не пользователю.

## Что не используется в первой версии

Не используются:

- `doctor-amal:requests`;
- `doctor-amal:storage-version`;
- `RequestRepository`;
- миграции пользовательского localStorage;
- экспорт заявок;
- сохранение форм.

Эти темы относятся к будущим версиям и требуют отдельных specs.

## Будущая замена на CMS или API

При переходе к онлайн-версии локальные данные проекта могут быть заменены на CMS, базу данных или API.

Первая версия должна быть написана так, чтобы UI получал данные через явный data/provider слой, а не смешивал содержимое страницы с JSX без структуры.
