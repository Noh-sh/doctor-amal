# Technical Spec: Локальное хранение

## Цель
Описать технический способ хранения данных на этапе локальной разработки и правила, которые позволят позже заменить локальное хранилище сервером.

## Данные первой версии
Первая версия использует три группы данных:
- курсы;
- статьи;
- заявки.

Курсы и статьи являются seed-данными. Заявки создаются пользователем.

## Рекомендуемое хранение для Next.js первой версии
Для Next.js проекта:
- `data/courses.json` или `src/data/courses.json` - seed-данные курсов;
- `data/articles.json` или `src/data/articles.json` - seed-данные статей;
- browser `localStorage` - заявки пользователя.

Ключи:

```text
doctor-amal:requests
doctor-amal:storage-version
```

В первой версии заявки не записываются в файл через Next.js server code. Это важно, чтобы локальная версия оставалась простой и не создавала ложного ощущения готовой серверной обработки заявок.

## Repository API
UI должен работать через repository/service functions.

```ts
interface CourseRepository {
  getPublishedCourses(): Promise<Course[]>;
  getCourseById(id: string): Promise<Course | null>;
}

interface ArticleRepository {
  getPublishedArticles(): Promise<Article[]>;
  getArticleById(id: string): Promise<Article | null>;
}

interface RequestRepository {
  listRequests(): Promise<UserRequest[]>;
  saveRequest(request: UserRequest): Promise<SaveRequestResult>;
}
```

Даже если реализация синхронная, API лучше оставить async, чтобы позже заменить его на серверный API.

В Next.js `CourseRepository` и `ArticleRepository` могут быть server-side modules. `RequestRepository` для первой версии должен быть client-side module, потому что использует browser `localStorage`.

## Загрузка курсов
При загрузке курсов система должна:
- прочитать локальный JSON;
- проверить, что верхний уровень является массивом;
- отфильтровать `publicationStatus !== "published"`;
- не показывать курсы с критически неполными данными;
- вернуть пустой массив при отсутствии опубликованных курсов.

Если JSON поврежден, система возвращает контролируемую ошибку и UI показывает понятное сообщение.

В Next.js чтение курсов может выполняться в Server Components или server repository. Клиентские компоненты не должны напрямую импортировать полный JSON, если это не требуется для конкретного интерактивного сценария.

## Загрузка статей
При загрузке статей система должна:
- прочитать локальный JSON;
- проверить, что верхний уровень является массивом;
- отфильтровать `publicationStatus !== "published"`;
- не показывать статьи без полного текста;
- сортировать по `publishedAt` от новых к старым, если дата указана.

В Next.js чтение статей может выполняться в Server Components или server repository.

## Сохранение заявок в localStorage
Алгоритм:
1. Прочитать ключ `doctor-amal:requests`.
2. Если ключа нет, считать текущий список пустым.
3. Если значение не является валидным JSON-массивом, не перезаписывать его автоматически.
4. Проверить повторную заявку.
5. Добавить новую заявку в массив.
6. Записать массив обратно.
7. Вернуть результат.

## Поврежденные данные
Если данные заявок в `localStorage` повреждены:
- не перезаписывать автоматически;
- вернуть ошибку `storage_unavailable` или отдельный internal error;
- показать пользователю понятный текст;
- в development можно вывести техническую причину в console, но не показывать ее пользователю.

## Миграции
Нужно хранить версию структуры:

```text
doctor-amal:storage-version = 1
```

В первой версии миграции могут быть простыми:
- если версии нет, считать ее `1`;
- если версия выше поддерживаемой, не пытаться читать данные как валидные;
- если в будущем структура изменится, добавить явную миграцию.

## Экспорт заявок
Экспорт заявок не обязателен для первой версии, но структура должна позволять добавить его позже.

Будущий экспорт должен:
- не включать платежные данные, потому что они не собираются;
- сохранять только поля заявки;
- быть доступен владельцу проекта, а не обычному пользователю публичного интерфейса.

## Нет прав на запись
В browser `localStorage` ошибка записи возможна при:
- приватном режиме;
- переполнении хранилища;
- ограничениях браузера.

UI должен показать:

```text
Заявку не удалось сохранить. Проверьте данные и попробуйте еще раз.
```

В первой Next.js версии запись в файл не используется. Если позже появится server-side сохранение, ошибки прав доступа должны обрабатываться на сервере и переводиться в понятный пользовательский текст.

## Принцип замены на сервер
Когда появится онлайн-версия:
- `CourseRepository` будет читать курсы из API;
- `ArticleRepository` будет читать статьи из API;
- `RequestRepository` будет отправлять заявки на сервер;
- UI и domain rules не должны измениться по смыслу.

Нельзя реализовывать UI так, чтобы компонент напрямую зависел от `localStorage`.

## Резервное поведение
Если курсы не загрузились:
- каталог показывает ошибку;
- приложение не падает.

Если статьи не загрузились:
- раздел статей показывает ошибку;
- остальные разделы работают.

Если заявки не сохраняются:
- форма остается заполненной;
- пользователь не получает ложного подтверждения.
