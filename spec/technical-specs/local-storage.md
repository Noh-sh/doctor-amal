# Technical Spec: Локальные данные и хранение

## Цель

Зафиксировать допустимый локальный fallback и запрет browser-хранения пользовательских данных в текущей online-версии.

## Главное правило

Текущая версия не собирает, не сохраняет и не обрабатывает данные публичного пользователя.

Поэтому в текущей версии не используется browser `localStorage` для заявок, обращений, контактов, медицинских данных или истории действий пользователя.

## Допустимые локальные данные проекта

В проекте могут храниться только данные для отображения страницы:

- профиль доктора;
- фото или путь к фото;
- короткое представление;
- текст блока `О докторе`;
- список курсов;
- описания курсов;
- цены или согласованные состояния без цены;
- внешние URL;
- Telegram-ссылка менеджера;
- текст медицинского ограничения;
- тексты неактивных состояний.

Эти данные служат fallback. Основной контент редактируется доктором через `/admin` и хранится в Supabase.

## Формат локального fallback

Локальный fallback хранится как TypeScript-константа с типом `TaplinkPageData`.

## Что запрещено хранить

В текущей версии нельзя хранить:

- имя пользователя;
- телефон;
- email;
- Telegram username пользователя;
- медицинские вопросы пользователя;
- медицинские документы;
- результаты анализов;
- заявки;
- заказы;
- платежные данные;
- историю кликов;
- аналитику.

## Поведение при неполных данных

Если данные неполные:

- неподтвержденный URL делает кнопку видимой, но неактивной;
- неподтвержденная Telegram-ссылка менеджера делает `Купить` неактивной;
- неподтвержденная цена не заменяется выдуманной суммой;
- неподтвержденный список курсов не заменяется выдуманными курсами;
- неподтвержденные данные о докторе не выводятся как факты.

## Поврежденные локальные данные проекта

Если локальный data-файл поврежден или не соответствует ожидаемой структуре:

- приложение не должно показывать белый экран;
- пользователь видит спокойное состояние или отсутствие неподтвержденного блока;
- техническая причина может быть видна разработчику в development, но не пользователю.

## Что не используется в текущей версии

Не используются:

- `doctor-amal:requests`;
- `doctor-amal:storage-version`;
- `RequestRepository`;
- миграции пользовательского localStorage;
- экспорт заявок;
- сохранение форм.

Эти темы относятся к будущим версиям и требуют отдельных specs.

## Supabase и fallback

Supabase является основным online-источником контента страницы. Локальные данные используются только при отсутствии env, неполных обязательных rows или ошибке чтения Supabase.

UI получает данные через явный data/provider слой и не знает, используется Supabase или fallback.

Локальный fallback применяется только для публичной страницы `/`.

Правила fallback:

- fallback содержит только заранее подтвержденный контент проекта;
- fallback не сохраняет и не читает данные публичного пользователя;
- fallback не используется для записи admin-изменений;
- если Supabase недоступен, публичная страница показывает fallback без технической ошибки для посетителя;
- если часть Supabase-данных отсутствует, недостающие внешние кнопки текущей версии остаются видимыми и неактивными;
- если fallback поврежден, приложение показывает спокойное пользовательское состояние без stack trace и без служебных деталей.

Админка `/admin` не должна считать fallback успешным подключением к Supabase. Если Supabase или Supabase Auth недоступны, формы редактирования не показываются до восстановления подключения и подтверждения доступа доктора.
