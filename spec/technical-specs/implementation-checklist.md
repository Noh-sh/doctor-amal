# Technical Spec: Implementation Checklist

## Цель
Дать проверочный список для реализации первой версии. По этому списку можно оценивать, готова ли функция технически и соответствует ли она feature specs.

## Архитектура
- Приложение реализовано на Next.js + TypeScript.
- Используется Next.js App Router.
- Приложение запускается локально через стандартный Next.js dev server.
- Основные сценарии работают без интернета.
- UI не читает JSON или `localStorage` напрямую, а использует repository/service layer.
- Компоненты, которые используют `localStorage` или browser state, помечены `"use client"`.
- Страницы, которые только читают seed-данные, могут быть Server Components.
- Бизнес-правила продаж вынесены в общие функции.
- Все пользовательские тексты на русском языке.
- Технические ошибки не показываются пользователю как stack trace.

## Данные
- Есть модель `Course`.
- Есть модель `Article`.
- Есть модель `UserRequest`.
- Курсы и статьи имеют стабильные строковые `id`.
- Черновики и архивные записи скрыты от пользователя.
- Заявки не содержат платежных данных, паролей, медицинских документов и анализов.

## Каталог курсов
- Показывает только опубликованные курсы.
- Показывает статус продаж.
- Поддерживает поиск по названию, теме, категории и симптомам.
- Поддерживает фильтр по теме или категории.
- Поддерживает фильтр по статусу продаж.
- При пустом списке показывает понятное пустое состояние.
- При поиске без результатов предлагает изменить запрос.
- Не предлагает покупку при закрытых продажах.

## Страница курса
- Открывает курс по `courseId`.
- Не показывает неопубликованный курс.
- Показывает формат доступа через закрытый Telegram-канал.
- Объясняет, что оплата не выполняется на сайте.
- При `open` показывает заявку на покупку.
- При `closed` не показывает заявку и предлагает статьи.
- При `coming_soon` не принимает заявку и показывает период, если он известен.
- Похожие курсы ведут на страницы курсов, а не сразу в заявку.

## Заявки
- Поддерживается `course_purchase`.
- Поддерживается `pre_purchase_consultation`.
- Общие поля валидируются.
- Поля покупки валидируются.
- Поля консультации валидируются.
- Согласие "не медицинская консультация" обязательно.
- Согласие "нет онлайн-оплаты" обязательно.
- Заявка на покупку проверяет статус продаж курса.
- Повторная похожая заявка за последние 10 минут блокируется или требует явного повторного действия.
- При ошибке сохранения форма не очищается.
- При успехе показывается корректное сообщение.

## Статьи
- Показываются только опубликованные статьи.
- Статьи без полного текста не открываются как полноценный материал.
- Есть список статей.
- Есть страница отдельной статьи.
- Статья не ведет напрямую к заявке на покупку.
- При закрытых продажах статьи остаются доступными.

## Страница "О докторе"
- Показывает публичную информацию о докторе и подходе.
- Объясняет роль курсов.
- Объясняет закрытый Telegram-формат.
- Объясняет отсутствие онлайн-оплаты.
- Показывает заявку на консультацию только при открытых продажах.
- Не обещает медицинский результат.

## Ошибки и пустые состояния
- Есть 404-страница.
- Есть сообщение "курс не найден".
- Есть сообщение "статья не найдена".
- Есть сообщение "курсы пока не опубликованы".
- Есть сообщение "материалы пока не опубликованы".
- Есть сообщение "заявку не удалось сохранить".
- Ошибки форм показываются рядом с полями.
- Ошибки не обвиняют пользователя.

## Медицинские и этические ограничения
- Приложение не ставит диагноз.
- Приложение не назначает лечение.
- Курсы описаны как образовательные материалы.
- Заявка на консультацию по подбору курса не описана как полноценная медицинская консультация.
- При опасных или острых симптомах пользователь направляется к очной медицинской помощи.

## Тестовые сценарии приемки

### Продажи открыты
1. Открыть каталог.
2. Найти курс со статусом `open`.
3. Открыть страницу курса.
4. Перейти к заявке на покупку.
5. Заполнить форму валидными данными.
6. Отправить заявку.
7. Увидеть сообщение об успехе.
8. Проверить, что заявка сохранена локально.

### Продажи закрыты
1. Открыть курс со статусом `closed`.
2. Проверить, что кнопки покупки нет.
3. Проверить, что заявка на консультацию не предлагается.
4. Перейти к статьям.

### Продажи скоро откроются
1. Открыть курс со статусом `coming_soon`.
2. Проверить, что заявка недоступна.
3. Проверить, что период продаж показан, если он заполнен.

### Невалидная форма
1. Открыть форму заявки.
2. Оставить имя и контакт пустыми.
3. Отправить форму.
4. Проверить, что заявка не сохранена.
5. Проверить, что ошибки показаны рядом с полями.

### Повторная заявка
1. Отправить валидную заявку.
2. Сразу отправить похожую заявку с тем же контактом и курсом.
3. Проверить, что система показывает предупреждение о похожей заявке.

### Поврежденные локальные данные
1. Повредить локальные данные заявок или seed JSON в тестовой среде.
2. Открыть соответствующую страницу.
3. Проверить, что приложение не падает.
4. Проверить, что пользователь видит понятную ошибку.

## Definition of Done
Функция считается готовой, если:
- реализованы основные сценарии из feature spec;
- выполнены ограничения из technical specs;
- есть обработка пустых и ошибочных состояний;
- данные проходят валидацию;
- недопустимые действия технически заблокированы;
- интерфейс не вводит пользователя в заблуждение относительно оплаты, медицины и Telegram-доступа;
- реализация не мешает будущей замене локального хранилища на сервер.
