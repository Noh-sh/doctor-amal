# Technical Spec: Архитектура

## Цель

Описать техническую архитектуру первой локальной Taplink-версии Doctor Amal так, чтобы реализация соответствовала specs и оставалась готовой к будущей онлайн-версии.

## Тип приложения

Первая версия — локально запускаемое web-приложение на Next.js.

Обязательные решения:

- Next.js App Router;
- TypeScript;
- React components;
- одна публичная страница `/`;
- локальные данные проекта;
- отсутствие server API, базы данных, авторизации, админки, форм заявок и онлайн-оплаты.

## Рекомендуемая структура проекта

```text
app/
  layout.tsx
  page.tsx
  not-found.tsx
components/
  taplink/
  common/
data/
  taplink-page.ts
domain/
  taplink.ts
lib/
  links/
    externalLinks.ts
  messages/
    ru.ts
styles/
  globals.css
```

Если проект использует папку `src`, такая же структура может находиться внутри `src/`.

Имена файлов могут отличаться, но разделение ответственности должно сохраняться:

- `app/` — маршруты и page-level компоненты Next.js;
- `components/` — UI-компоненты страницы;
- `data/` — локальные данные проекта;
- `domain/` — типы и правила;
- `lib/` — helper-функции и подготовка ссылок;
- `styles/` — стили.

## Server и Client Components

По умолчанию публичная страница может быть Server Component.

Client Components нужны только для интерактивного поведения:

- раскрытие курса;
- раскрытие блока, если выбран такой UI-вариант;
- управление активным состоянием accordion;
- обработка клика, если нужна client-side логика.

Компоненты с browser state, событиями, accordion или другим интерактивным состоянием должны быть помечены `"use client"`.

В первой версии не должно быть client-side чтения или записи пользовательских данных в `localStorage`.

## Слои приложения

### UI layer

Отвечает за:

- отображение одной страницы;
- первый блок с фото, именем и кнопками;
- блок `О докторе`;
- блок `Курсы`;
- внешние кнопки;
- кнопку `Купить`;
- неактивные состояния;
- адаптивность и доступность.

UI layer не должен придумывать данные, URL, цены или медицинские формулировки.

### Domain layer

Отвечает за:

- типы `DoctorProfile`, `Course`, `ExternalLink`, `TaplinkPageData`;
- правила активности внешних ссылок;
- правила активности кнопки `Купить`;
- подготовку общей Telegram-ссылки менеджера без автоматической передачи названия курса в первой версии.

### Data layer

Отвечает за:

- локальные данные профиля;
- локальные данные курсов;
- внешние URL;
- общую Telegram-ссылку менеджера;
- короткое медицинское ограничение.

Данные можно хранить в TypeScript-константах или JSON. Для первой версии предпочтительнее TypeScript-константы с типизацией, если это упрощает проверку структуры.

## Главные технические ограничения

Приложение не должно:

- создавать внутренние маршруты для `О докторе`, `Курсы`, курса, покупки, заявки или консультации;
- собирать данные пользователя;
- сохранять обращения пользователя;
- создавать заказ;
- принимать оплату;
- подключать платежного провайдера;
- автоматически выдавать доступ в Telegram;
- автоматически передавать название курса в Telegram без новых specs;
- показывать неподтвержденные URL как рабочие ссылки;
- показывать придуманные курсы, цены или тексты.

## Подготовка к будущей онлайн-версии

Код должен быть организован так, чтобы позже можно было заменить локальные данные на CMS/API без переписывания пользовательского сценария.

Рекомендуется:

- держать данные отдельно от UI;
- использовать типы для структуры данных;
- формировать внешние URL через helper-функции;
- не смешивать первую локальную версию с будущей админкой или онлайн-покупкой.

В первой версии не реализуются:

- `app/api/*`;
- admin routes;
- database client;
- authentication;
- server actions для сохранения данных;
- платежные интеграции.

## Локализация

Все пользовательские тексты — на русском языке.

Технические имена файлов, типов и полей могут быть на английском.

## Производительность

Объем данных небольшой. Не нужны:

- серверная пагинация;
- виртуализация списков;
- сложная client-side фильтрация;
- prefetch большого количества данных.

Страница должна быстро открываться на мобильных устройствах и не выполнять лишнюю клиентскую работу.

## Доступность

Обязательные требования:

- интерактивные элементы доступны с клавиатуры;
- кнопки имеют понятный текст или доступное имя;
- неактивные кнопки визуально и семантически отличаются от активных;
- цвет не является единственным способом показать статус;
- раскрытие курса должно быть доступно с клавиатуры;
- фокус не должен теряться при раскрытии курса.

## Нефункциональные требования

- Dev server запускается стандартной командой Next.js, например `npm run dev`.
- Production build собирается стандартной командой Next.js, например `npm run build`.
- Ошибки локальных данных не должны приводить к белому экрану.
- Пользователь не должен видеть stack trace или технические идентификаторы.
- Ограничения проекта должны быть отражены в UI, а не только в коде.
