# Technical Spec: Архитектура

## Цель
Описать техническую архитектуру первой локальной версии Doctor-Amal так, чтобы реализация соответствовала продуктовым спецификациям и могла быть позже перенесена в онлайн-версию.

## Тип приложения
Первая версия должна быть локально запускаемым web-приложением на Next.js.

Минимально допустимый вариант:
- Next.js App Router;
- работает в браузере локально через Next.js dev/build server;
- данные курсов и статей загружаются из локальных JSON;
- заявки сохраняются локально.

## Рекомендуемая структура проекта
При реализации на Next.js + TypeScript рекомендуется структура:

```text
app/
  layout.tsx
  page.tsx
  doctor/
    page.tsx
  courses/
    page.tsx
    [courseId]/
      page.tsx
  request/
    page.tsx
  articles/
    page.tsx
    [articleId]/
      page.tsx
  not-found.tsx
components/
  layout/
  courses/
  articles/
  requests/
  common/
data/
  courses.json
  articles.json
domain/
  course.ts
  article.ts
  request.ts
  sales.ts
lib/
  repositories/
    courseRepository.ts
    articleRepository.ts
    requestRepository.client.ts
  validation/
    requestValidation.ts
  messages/
    ru.ts
styles/
  globals.css
```

Если проект создается с папкой `src`, допустим такой же вариант внутри `src/`:

```text
src/
  app/
    layout.tsx
    page.tsx
    ...
  components/
  data/
  domain/
  lib/
  styles/
```

Эта структура не обязательна по именам файлов, но обязательна по разделению ответственности.

## Server и Client Components
Next.js реализация должна явно разделять server и client responsibilities.

Server Components подходят для:
- страниц каталога;
- страницы курса;
- списка статей;
- страницы статьи;
- чтения локальных JSON seed-данных;
- подготовки SEO metadata, если она будет добавлена.

Client Components нужны для:
- интерактивного поиска и фильтров, если они работают без перезагрузки страницы;
- формы заявки;
- чтения и записи `localStorage`;
- блокировки кнопки во время отправки;
- показа ошибок валидации после пользовательского ввода.

Компоненты с `localStorage`, browser events или form state должны быть помечены `"use client"`.

## Слои приложения

### UI layer
Отвечает за:
- отображение страниц;
- формы;
- состояния загрузки, пустых списков и ошибок;
- навигацию;
- показ доступных действий в зависимости от статуса продаж.

UI layer не должен напрямую решать бизнес-правила продаж и валидации. Эти правила должны быть вынесены в domain/services.

### Domain layer
Отвечает за:
- типы и модели данных;
- статусы курсов, заявок и публикаций;
- правила доступности покупки;
- правила показа заявки на консультацию;
- нормализацию поисковых запросов и фильтров.

### Repository/service layer
Отвечает за:
- загрузку курсов;
- загрузку статей;
- сохранение заявок;
- чтение ранее сохраненных заявок, если это нужно для проверки повторной отправки;
- обработку технических ошибок чтения/записи.

### Data layer
Отвечает за:
- seed-данные курсов;
- seed-данные статей;
- локально сохраненные заявки.

В Next.js seed-данные курсов и статей могут импортироваться из JSON в server-side repository. Заявки в первой локальной версии сохраняются только на клиенте в `localStorage`, поэтому repository для заявок должен быть client-only.

## Обязательные страницы
Первая версия должна поддерживать:
- главная страница;
- страница "О докторе";
- каталог курсов;
- страница курса;
- форма заявки;
- список статей;
- страница статьи;
- страница ошибки 404.

## Главные бизнес-правила
- Покупка курса доступна только при статусе продаж `open`.
- При статусе `closed` заявка на покупку и заявка на консультацию по подбору курса не предлагаются.
- При статусе `coming_soon` заявки не принимаются.
- Оплата не выполняется в приложении.
- Доступ в закрытый Telegram-канал предоставляется вручную вне приложения.
- Пользователь не видит неопубликованные курсы и статьи.
- Пользователь не видит внутренние идентификаторы, названия закрытых каналов и служебные поля.

## Offline-first поведение
Приложение должно оставаться работоспособным без интернета:
- страницы открываются;
- каталог и статьи читаются из локальных данных;
- формы валидируются локально;
- заявки сохраняются локально;
- внешние ссылки на Telegram могут быть недоступны, но это не должно ломать приложение.

Если ссылка на Telegram не открывается из-за отсутствия интернета, приложение не должно считать это критической ошибкой основной функциональности.

## Подготовка к будущей онлайн-версии
Даже если первая версия хранит данные локально, код должен быть организован так, чтобы позже заменить repository layer на API-клиент.

Запрещено жестко связывать UI с конкретным способом хранения. Например, страница каталога должна вызывать `getPublishedCourses()`, а не читать JSON напрямую внутри компонента.

Для будущей онлайн-версии Next.js структура должна позволять добавить:
- Route Handlers в `app/api/*`;
- Server Actions, если они будут выбраны для заявок;
- подключение базы данных;
- административные страницы;
- серверную отправку уведомлений доктору или ассистенту.

В первой версии эти возможности не реализуются.

## Безопасность и приватность
Первая версия не должна собирать лишние персональные данные.

Обязательные правила:
- не хранить платежные данные;
- не хранить пароли;
- не хранить медицинские документы;
- не хранить результаты анализов;
- не логировать содержимое заявок в публичную консоль в production-сборке;
- не показывать технические ошибки пользователю без понятного пользовательского текста.

## Локализация
В первой версии весь интерфейс на русском языке.

Тексты ошибок, статусов и кнопок должны быть собраны в одном месте или в небольшом наборе констант, чтобы не было разной терминологии на разных страницах.

## Производительность
Ожидаемый объем данных небольшой, поэтому серверная пагинация не нужна.

При этом приложение должно:
- не выполнять тяжелую фильтрацию при каждом рендере без необходимости;
- не блокировать интерфейс при ошибках чтения;
- корректно показывать пустые состояния.

## Доступность
Формы и навигация должны:
- иметь видимые labels;
- поддерживать управление клавиатурой;
- показывать ошибки рядом с соответствующими полями;
- не полагаться только на цвет для передачи статуса;
- иметь понятные тексты кнопок.

## Нефункциональные требования
- Основные сценарии не требуют авторизации.
- Приложение не зависит от онлайн-сервера.
- Dev server должен запускаться стандартной командой Next.js, например `npm run dev`.
- Production build должен собираться стандартной командой Next.js, например `npm run build`.
- Ошибки данных не должны приводить к белому экрану.
- Все ограничения проекта должны быть отражены в UI, а не только в коде.
