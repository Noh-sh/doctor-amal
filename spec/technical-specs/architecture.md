# Technical Spec: Архитектура

## Цель

Описать техническую архитектуру текущей online Taplink-версии Doctor Amal.

## Тип приложения

Текущая Taplink-версия — web-приложение на Next.js.

Обязательные решения:

- Next.js App Router;
- TypeScript;
- React components;
- одна публичная страница `/`;
- защищенная админка `/admin`;
- Supabase как основной источник контента;
- локальные данные как fallback публичной страницы;
- Supabase Auth только для доктора;
- отсутствие auth покупателей, форм заявок и онлайн-оплаты.

## Рекомендуемая структура проекта

```text
app/
  layout.tsx
  page.tsx
  not-found.tsx
components/
  taplink/
  admin/
data/
  taplink-page.ts
  taplink-page-source.ts
domain/
  taplink.ts
lib/
  links/
    externalLinks.ts
  supabase/
styles/
  globals.css
supabase/
  migrations/
```

Если проект использует папку `src`, такая же структура может находиться внутри `src/`.

Имена файлов могут отличаться, но разделение ответственности должно сохраняться:

- `app/` — маршруты и page-level компоненты Next.js;
- `components/` — UI-компоненты страницы;
- `data/` — локальные данные проекта;
- `domain/` — типы и правила;
- `lib/` — helper-функции и подготовка ссылок;
- `styles/` — стили.

## Server и Client Components

По умолчанию публичная страница может быть Server Component.

Публичные accordion-блоки реализованы через native `details` / `summary` и не требуют Client Component.

Админка является Client Component, потому что использует Supabase Auth, browser session, формы и события.

В текущей версии не должно быть client-side чтения или записи пользовательских данных в `localStorage`.

## Слои приложения

### UI layer

Отвечает за:

- отображение одной страницы;
- первый блок с фото, именем и кнопками;
- блок `О докторе`;
- блок `Курсы`;
- внешние кнопки;
- кнопку `Купить`;
- неактивные состояния;
- адаптивность и доступность.

UI layer не должен придумывать данные, URL, цены или медицинские формулировки.

### Domain layer

Отвечает за:

- типы `DoctorProfile`, `Course`, `ExternalLink`, `TaplinkPageData`;
- правила активности внешних ссылок;
- правила активности кнопки `Купить`;
- подготовку общей Telegram-ссылки менеджера без автоматической передачи названия курса в текущей версии.

### Data layer

Отвечает за:

- чтение публичного контента из Supabase;
- преобразование rows в `TaplinkPageData`;
- локальный fallback при отсутствии или недоступности Supabase;
- чтение и запись разрешенного контента через админку.

UI продолжает получать `TaplinkPageData` и не знает, пришли данные из Supabase или локального fallback.

Для публичной страницы data layer должен быть устойчивым к ожидаемым сбоям Supabase:

- отсутствующие env-переменные Supabase;
- сетевые ошибки чтения;
- недоступность Supabase project;
- отсутствие обязательных опубликованных rows;
- частичные данные по внешним ссылкам, курсам или настройкам покупки.

В этих случаях публичная страница использует подтвержденный локальный fallback или безопасные неактивные состояния. Пользователь не должен видеть stack trace, SQL/PostgREST детали, названия policies, env-ключи или внутренние идентификаторы.

Для админки data layer не должен подменять Supabase локальным fallback при записи. Если Supabase или Supabase Auth недоступны, `/admin` показывает понятное русское сообщение об ошибке и не показывает формы редактирования до восстановления подключения и подтверждения роли `doctor_admin`.

## Главные технические ограничения

Приложение не должно:

- создавать внутренние маршруты для `О докторе`, `Курсы`, курса, покупки, заявки или консультации;
- собирать данные пользователя;
- сохранять обращения пользователя;
- создавать заказ;
- принимать оплату;
- подключать платежного провайдера;
- автоматически выдавать доступ в Telegram;
- автоматически передавать название курса в Telegram без новых specs;
- показывать неподтвержденные URL как рабочие ссылки;
- показывать придуманные курсы, цены или тексты.

## Текущие ограничения

Не реализуются:

- `app/api/*`;
- auth покупателей;
- управление Auth users через админку;
- server actions с service role key;
- платежные интеграции;
- загрузка фото;
- физическое удаление курсов.

## Локализация

Все пользовательские тексты — на русском языке.

Технические имена файлов, типов и полей могут быть на английском.

## Производительность

Объем данных небольшой. Не нужны:

- серверная пагинация;
- виртуализация списков;
- сложная client-side фильтрация;
- prefetch большого количества данных.

Страница должна быстро открываться на мобильных устройствах и не выполнять лишнюю клиентскую работу.

## Доступность

Обязательные требования:

- интерактивные элементы доступны с клавиатуры;
- кнопки имеют понятный текст или доступное имя;
- неактивные кнопки визуально и семантически отличаются от активных;
- цвет не является единственным способом показать статус;
- раскрытие курса должно быть доступно с клавиатуры;
- фокус не должен теряться при раскрытии курса.

## Нефункциональные требования

- Dev server запускается стандартной командой Next.js, например `npm run dev`.
- Production build собирается стандартной командой Next.js, например `npm run build`.
- Ошибки локальных данных не должны приводить к белому экрану.
- Ошибки чтения Supabase на публичной странице не должны приводить к белому экрану, если доступен локальный fallback.
- Недоступность Supabase/Auth в админке не должна открывать формы редактирования.
- Пользователь не должен видеть stack trace или технические идентификаторы.
- Ограничения проекта должны быть отражены в UI, а не только в коде.
