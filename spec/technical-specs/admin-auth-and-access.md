# Technical Spec: Admin Auth и доступ к редактированию контента

## Цель

Описать технические требования текущей админки Doctor Amal, через которую доктор редактирует подтвержденный контент страницы.

Этот spec описывает реализованную модель доступа. Изменения модели требуют отдельного spec, migration и work plan.

## Граница этапа

Входит:

- Supabase Auth для входа доктора;
- восстановление пароля доктора через Supabase Auth email flow;
- защищенный маршрут `/admin`;
- ограничение редактирования только для доктора;
- запись изменений в таблицы контента Supabase;
- загрузка и замена фото доктора через Supabase Storage;
- безопасное удаление курса через soft delete;
- RLS policies для безопасной записи;
- сохранение текущего публичного сценария `/`.

Не входит:

- auth покупателей;
- регистрация пользователей;
- личный кабинет покупателя;
- заявки;
- заказы;
- онлайн-оплата;
- платежные статусы;
- медицинские анкеты;
- хранение персональных медицинских данных;
- аналитика;
- рассылки;
- роль менеджера на первом этапе;
- журнал изменений;
- черновики и предпросмотр публикации.

## Модель входа

Подтвержденный способ входа:

```text
email/password
```

Пользователь Auth создается только для доктора.

Публичные пользователи сайта не регистрируются и не входят в систему.

## Восстановление пароля

Доктор должен иметь простой способ восстановить пароль без доступа к Supabase Dashboard.

На форме входа `/admin` должна быть доступна команда:

```text
Забыли пароль?
```

Подтвержденный сценарий:

1. Доктор открывает `/admin`.
2. Доктор нажимает `Забыли пароль?`.
3. Админка показывает форму ввода email для восстановления пароля.
4. Доктор вводит email своего Supabase Auth user.
5. Админка отправляет recovery email через Supabase Auth.
6. Интерфейс показывает нейтральное сообщение:

```text
Если email зарегистрирован, ссылка для восстановления отправлена.
```

7. Доктор открывает ссылку из письма.
8. Доктор попадает на экран задания нового пароля.
9. После успешной смены пароля доктор возвращается к входу в `/admin`.

Recovery flow не должен сообщать, существует ли введенный email. Это нужно, чтобы не раскрывать Auth users.

Recovery flow не создает:

- регистрацию покупателей;
- личный кабинет;
- публичный пользовательский вход;
- новые роли;
- управление Auth users через UI;
- доступ к Supabase Dashboard для доктора.

Технические требования:

- recovery email отправляется через Supabase Auth;
- redirect URL recovery flow должен вести только на подтвержденный route текущего приложения;
- экран нового пароля должен проверять, что новый пароль не пустой;
- после успешной смены пароля сессия восстановления не должна показывать формы редактирования, пока роль `doctor_admin` не подтверждена обычной проверкой доступа;
- технические ошибки показываются понятным русским текстом без stack trace.

Безопасность:

- пароль не хранится в таблицах приложения;
- recovery token, recovery URL, email доктора, UUID Auth user и любые секреты нельзя записывать в git, specs, work plans, roadmap или комментарии к коду;
- service role key не нужен для recovery UI и не должен использоваться во frontend-коде.

## Роли

На первом этапе есть три технических состояния:

- `anon` — обычный неавторизованный посетитель публичной страницы;
- `authenticated` без права администратора — авторизованный пользователь без доступа к редактированию;
- `doctor_admin` — доктор, которому разрешено редактировать контент.

Роль `doctor_admin` должна определяться не только фактом входа, а отдельной проверкой права доступа.

## Таблица доступа

Для контроля админ-доступа используется таблица:

```text
admin_users
```

Обязательные поля:

- `user_id` — UUID пользователя из Supabase Auth;
- `role` — техническая роль, на первом этапе `doctor_admin`;
- `is_active` — активен ли доступ;
- `created_at` — техническая дата создания;
- `updated_at` — техническая дата обновления.

Эта таблица не хранит покупателей и не используется для публичных пользователей.

Таблица создается migration `20260603000000_create_admin_access_policies.sql`.

## RLS и доступ

RLS должен оставаться включенным на таблицах контента.

Публичные `select` policies первого Supabase-этапа сохраняются:

- публичная страница читает только опубликованный или активный контент;
- публичный anon key не получает write-доступ.

Write policies разрешают изменение контента только активному `doctor_admin`.

Минимально допустимые операции для `doctor_admin`:

- `update` для `doctor_profile`;
- `update` для `external_links`;
- `update` для `courses`;
- `insert` для `courses`;
- `update` для `purchase_settings`.
- upload/update для фото в Storage bucket текущего проекта.

На первом этапе `page_settings.medical_notice` не редактируется через админку.

`insert` policy для `courses` разрешает создание курса только активному `doctor_admin`.

`anon` не должен получать write-доступ к `courses`.

Физический `delete` для `courses` отсутствует. Курс скрывается через `is_active = false` или безопасно удаляется через soft delete, например `deleted_at`.

## Защищенный маршрут

Текущий маршрут:

```text
/admin
```

Требования:

- без сессии пользователь видит форму входа;
- на форме входа есть доступ к восстановлению пароля;
- пользователь без роли `doctor_admin` не видит формы редактирования;
- если Supabase Auth или проверка `admin_users` недоступны, пользователь не видит формы редактирования;
- при недоступности Supabase/Auth админка показывает понятный русский feedback без stack trace, SQL/PostgREST деталей, recovery tokens, email или UUID Auth user;
- публичная страница `/` не показывает ссылку на админку как пользовательское действие;
- ошибки доступа показываются понятным русским текстом;
- технические stack trace не показываются пользователю.

Локальный fallback публичной страницы не используется для админки как источник прав доступа или источник данных для сохранения. Редактирование доступно только после успешной проверки Supabase Auth session и активной роли `doctor_admin`.

## Данные, которые можно менять

Админка может менять только поля, нужные для публичного контента:

- текстовые поля `doctor_profile`;
- `doctor_profile.photo_src`;
- `doctor_profile.photo_alt`;
- `external_links.url`;
- `external_links.is_enabled`;
- `external_links.inactive_text`;
- `courses.title`;
- `courses.description`;
- `courses.price_display_text`;
- `courses.price_is_confirmed`;
- `courses.is_active`;
- техническое поле безопасного удаления курса, например `courses.deleted_at`;
- `purchase_settings.manager_telegram_url`;
- `purchase_settings.inactive_text`.

При создании нового курса админка может задавать только:

- `courses.title`;
- `courses.description`;
- `courses.price_display_text`;
- `courses.price_is_confirmed`;
- `courses.is_active`.

Технические поля нового курса формируются приложением:

- `courses.id`;
- `courses.slug`;
- `courses.purchase_label`;
- `courses.sort_order`.

Админка не должна менять:

- технические id без отдельного требования;
- RLS policies;
- Auth users;
- env-переменные;
- migration-файлы;
- `page_settings.medical_notice` на первом этапе;
- поля, связанные с оплатой, заказами или покупателями.

## Создание курса

Админка может создавать новый курс только для публичного блока `Курсы`.

Обязательные поля при создании:

- название курса;
- описание курса;
- цена текстом.

Дополнительные поля:

- подтверждена ли цена;
- показывать ли курс на публичной странице.

Правила технических полей:

- `id` создается приложением как стабильный технический ключ;
- `slug` создается приложением из названия курса или из технического `id`;
- `purchase_label` всегда получает значение `Купить`;
- `sort_order` получает значение после последнего существующего курса;
- `is_active` определяет, показывается ли курс на публичной странице.

Если новый курс активен, он сохраняется в Supabase сразу после успешного сохранения. После сохранения приложение должно выполнить защищенный server-side revalidation публичной страницы `/`, чтобы курс появился сразу или почти сразу.

Если новый курс неактивен, он остается в админке, но публичная страница его не показывает.

Создание курса не добавляет:

- отдельную страницу курса;
- отдельную Telegram-ссылку курса;
- заявку;
- оплату;
- покупателя;
- доступ к закрытым материалам.

## Скрытие курса

Если доктор хочет убрать курс с публичной страницы, админка меняет `courses.is_active` на `false`.

Скрытый курс не должен отображаться на публичной странице и не должен оставлять пустой блок в списке.

Скрытый курс остается в Supabase и может быть снова включен через админку.

## Безопасное удаление курса

Если доктор хочет удалить курс из основного списка админки и публичной страницы, админка выполняет soft delete.

Минимальное техническое правило:

- запись курса остается в Supabase;
- у курса заполняется технический признак удаления, например `deleted_at`;
- публичная страница не читает удаленные курсы;
- основной список курсов в админке не показывает удаленные курсы;
- физический SQL `delete` через UI не используется.

Перед soft delete админка должна показать предупреждение и запросить подтверждение действия.

Удаленный курс не должен отображаться на публичной странице и не должен оставлять пустой блок.

Восстановление удаленного курса через UI не входит в этот этап. При ошибочном удалении восстановление возможно только технически через Supabase после отдельного решения владельца проекта.

Физическое удаление курса через `delete` на этом этапе не реализуется.

Для физического удаления нужна отдельная спецификация, отдельная `delete` policy и отдельная проверка восстановления после случайного удаления.

## Фото доктора

Таблица `doctor_profile` содержит `photo_src` и `photo_alt`.

В новом admin-этапе доктор может загрузить или заменить фото через `/admin`.

Требования:

- фото загружает только активный `doctor_admin`;
- файл хранится в Supabase Storage;
- допустимы JPEG, PNG и WebP;
- максимальный размер файла `5 MB`;
- после успешной загрузки обновляются `doctor_profile.photo_src` и `doctor_profile.photo_alt`;
- `photo_alt` должен быть русскоязычным и пользовательским, например `Фото: {имя доктора}`;
- service role key не используется во frontend;
- ошибки Storage или сохранения показываются без stack trace и служебных деталей.

Если загрузка файла прошла, но обновление `doctor_profile` не удалось, публичная страница не должна получать битую ссылку как актуальное фото.

## Публикация и кеш

На первом этапе изменения публикуются сразу после сохранения.

Публичная страница использует `revalidate: 30` как обычный fallback-механизм.

После успешного сохранения через `/admin` приложение должно инициировать server-side revalidation публичного маршрута `/`.

Требования к revalidation:

- вызов выполняется только после успешной записи в Supabase;
- вызов доступен только пользователю с подтвержденной Supabase Auth session и активной ролью `doctor_admin`;
- обычный `anon` посетитель и `authenticated` пользователь без `doctor_admin` не могут сбрасывать кеш;
- frontend не использует service role key;
- route/action revalidation не меняет данные Supabase и не обходит RLS;
- при ошибке revalidation сохранение не откатывается, но админка показывает понятный feedback.

Если Supabase недоступен во время сохранения, изменение не считается опубликованным. Админка должна показать ошибку сохранения без технических деталей и не обещать, что публичная страница обновится.

## Валидация

Админка должна проверять:

- обязательные текстовые поля не пустые;
- фото имеет допустимый тип и размер до загрузки;
- активная внешняя ссылка имеет URL;
- если Telegram URL отсутствует, задан текст неактивной кнопки;
- цена курса хранится как текст;
- при создании курса название, описание и цена не пустые;
- описание курса не должно явно нарушать medical content rules;
- кнопка покупки остается `Купить`.

Медицинский смысл текста должен проверяться по `spec/feature-specs/medical-content-rules.md`.

## Безопасность

Запрещено:

- использовать service role key во frontend-коде;
- добавлять secret key в `NEXT_PUBLIC_*`;
- добавлять `DOCTOR_SUPABASE_SERVICE_ROLE_KEY` в browser/client Supabase;
- давать write-доступ роли `anon`;
- давать write-доступ любому `authenticated` пользователю без проверки `admin_users`;
- хранить пароль доктора в таблицах приложения;
- записывать recovery-ссылки или recovery tokens в specs, work plans, roadmap, git history или логи проекта;
- хранить ключи или токены в specs, work plans, roadmap или комментариях к коду.

## Проверка текущей реализации

Минимальные проверки после реализации:

```bash
npm run build
git diff --check
```

Ручные проверки:

- обычный посетитель открывает `/` без авторизации;
- обычный посетитель не может редактировать контент;
- доктор может войти через email/password;
- доктор может запросить письмо восстановления пароля без доступа к Supabase Dashboard;
- recovery form показывает нейтральное сообщение и не раскрывает наличие email;
- доктор может задать новый пароль после перехода по recovery link;
- при недоступном Supabase/Auth админка показывает понятный feedback и не показывает формы редактирования;
- доктор может сохранить цену курса;
- доктор может создать новый курс с названием, описанием и ценой;
- новый активный курс появляется на публичной странице сразу или почти сразу после успешного сохранения и revalidation;
- если revalidation не сработал, доктор видит понятный feedback, а публичная страница может обновиться по обычному `revalidate: 30`;
- новый неактивный курс не появляется на публичной странице;
- скрытый курс не оставляет пустое место на публичной странице;
- доктор может включить или выключить курс;
- доктор может изменить внешнюю ссылку;
- публичная страница показывает обновленные данные;
- публичный anon key не имеет write-доступа;
- service role key не используется во frontend.
- доктор может загрузить или заменить фото допустимого формата;
- недопустимый тип или слишком большой файл фото не сохраняется;
- после замены фото публичная страница обновляется сразу или почти сразу;
- доктор может удалить курс через soft delete после подтверждения;
- удаленный курс исчезает из публичной страницы и основного списка админки;
- физический SQL `delete` через UI не выполняется.

## Реализованные migrations

- `20260603000000_create_admin_access_policies.sql` — `admin_users`, `is_doctor_admin()` и update policies;
- `20260603001000_restrict_admin_update_grants.sql` — ограниченные grants;
- `20260603002000_add_admin_content_select_policies.sql` — чтение контента активным `doctor_admin`;
- `20260603003000_add_admin_course_insert_policy.sql` — создание курса активным `doctor_admin`.

Любое изменение прав доступа требует новой migration и проверки RLS через CLI/MCP.
