# Technical Spec: Admin Auth и доступ к редактированию контента

## Цель

Описать технические требования для будущей админки Doctor Amal, через которую доктор сможет редактировать подтвержденный контент страницы.

Этот spec является подготовкой к реализации. Он не включает немедленное изменение кода, Supabase Auth users, RLS policies или migrations.

## Граница этапа

Входит:

- Supabase Auth для входа доктора;
- защищенный маршрут будущей админки;
- ограничение редактирования только для доктора;
- запись изменений в таблицы контента Supabase;
- RLS policies для безопасной записи;
- сохранение текущего публичного сценария `/`.

Не входит:

- auth покупателей;
- регистрация пользователей;
- личный кабинет покупателя;
- заявки;
- заказы;
- онлайн-оплата;
- платежные статусы;
- медицинские анкеты;
- хранение персональных медицинских данных;
- аналитика;
- рассылки;
- роль менеджера на первом этапе;
- журнал изменений;
- черновики и предпросмотр публикации.

## Модель входа

Подтвержденный способ входа:

```text
email/password
```

Пользователь Auth создается только для доктора.

Публичные пользователи сайта не регистрируются и не входят в систему.

## Роли

На первом этапе есть три технических состояния:

- `anon` — обычный неавторизованный посетитель публичной страницы;
- `authenticated` без права администратора — авторизованный пользователь без доступа к редактированию;
- `doctor_admin` — доктор, которому разрешено редактировать контент.

Роль `doctor_admin` должна определяться не только фактом входа, а отдельной проверкой права доступа.

## Рекомендуемая таблица доступа

Для контроля админ-доступа рекомендуется отдельная таблица:

```text
admin_users
```

Предварительные поля:

- `user_id` — UUID пользователя из Supabase Auth;
- `role` — техническая роль, на первом этапе `doctor_admin`;
- `is_active` — активен ли доступ;
- `created_at` — техническая дата создания;
- `updated_at` — техническая дата обновления.

Эта таблица не хранит покупателей и не используется для публичных пользователей.

Создание этой таблицы требует отдельной migration перед реализацией.

## RLS и доступ

RLS должен оставаться включенным на таблицах контента.

Публичные `select` policies первого Supabase-этапа сохраняются:

- публичная страница читает только опубликованный или активный контент;
- публичный anon key не получает write-доступ.

Для админки нужно добавить write policies, которые разрешают изменение контента только активному `doctor_admin`.

Минимально допустимые операции для `doctor_admin`:

- `update` для `doctor_profile`;
- `update` для `external_links`;
- `update` для `courses`;
- `insert` для `courses` после отдельной migration добавления курса;
- `update` для `purchase_settings`.

На первом этапе `page_settings.medical_notice` не редактируется через админку.

Для добавления новых курсов через админку нужно добавить отдельную `insert` policy для `courses`.

`insert` policy должна разрешать создание курса только активному `doctor_admin`.

`anon` не должен получать write-доступ к `courses`.

Если позже потребуется удаление курсов, нужно отдельно подтвердить `delete` policy. По умолчанию предпочтительнее скрывать курс через `is_active = false`, а не удалять.

На текущем этапе `delete` для `courses` не добавляется.

## Защищенный маршрут

Будущий маршрут:

```text
/admin
```

Требования:

- без сессии пользователь видит только форму входа или перенаправляется на вход;
- пользователь без роли `doctor_admin` не видит формы редактирования;
- публичная страница `/` не показывает ссылку на админку как пользовательское действие;
- ошибки доступа показываются понятным русским текстом;
- технические stack trace не показываются пользователю.

## Данные, которые можно менять

Админка может менять только поля, нужные для публичного контента:

- текстовые поля `doctor_profile`;
- `external_links.url`;
- `external_links.is_enabled`;
- `external_links.inactive_text`;
- `courses.title`;
- `courses.description`;
- `courses.price_display_text`;
- `courses.price_is_confirmed`;
- `courses.is_active`;
- `purchase_settings.manager_telegram_url`;
- `purchase_settings.inactive_text`.

При создании нового курса админка может задавать только:

- `courses.title`;
- `courses.description`;
- `courses.price_display_text`;
- `courses.price_is_confirmed`;
- `courses.is_active`.

Технические поля нового курса формируются приложением:

- `courses.id`;
- `courses.slug`;
- `courses.purchase_label`;
- `courses.sort_order`.

Админка не должна менять:

- технические id без отдельного требования;
- RLS policies;
- Auth users;
- env-переменные;
- migration-файлы;
- `page_settings.medical_notice` на первом этапе;
- поля, связанные с оплатой, заказами или покупателями.

## Создание курса

Админка может создавать новый курс только для публичного блока `Курсы`.

Обязательные поля при создании:

- название курса;
- описание курса;
- цена текстом.

Дополнительные поля:

- подтверждена ли цена;
- показывать ли курс на публичной странице.

Правила технических полей:

- `id` создается приложением как стабильный технический ключ;
- `slug` создается приложением из названия курса или из технического `id`;
- `purchase_label` всегда получает значение `Купить`;
- `sort_order` получает значение после последнего существующего курса;
- `is_active` определяет, показывается ли курс на публичной странице.

Если новый курс активен, публичная страница показывает его после сохранения и обновления кеша.

Если новый курс неактивен, он остается в админке, но публичная страница его не показывает.

Создание курса не добавляет:

- отдельную страницу курса;
- отдельную Telegram-ссылку курса;
- заявку;
- оплату;
- покупателя;
- доступ к закрытым материалам.

## Скрытие курса

Если доктор хочет убрать курс с публичной страницы, админка меняет `courses.is_active` на `false`.

Скрытый курс не должен отображаться на публичной странице и не должен оставлять пустой блок в списке.

Скрытый курс остается в Supabase и может быть снова включен через админку.

Физическое удаление курса через `delete` на этом этапе не реализуется.

Для физического удаления нужна отдельная спецификация, отдельная `delete` policy и отдельная проверка восстановления после случайного удаления.

## Фото доктора

Текущая таблица `doctor_profile` содержит `photo_src` и `photo_alt`.

Редактирование фото через админку допускается только после отдельного технического уточнения:

- используется ли Supabase Storage или внешний URL;
- кто может загружать файл;
- какие MIME-типы разрешены;
- какой максимальный размер файла;
- как работает preview;
- как заменяется старый файл;
- как защищается storage bucket;
- как не допустить загрузку неподтвержденного изображения.

До этого админка может показывать текущее фото, но не обязана давать загрузку нового файла.

## Публикация и кеш

На первом этапе изменения публикуются сразу после сохранения.

Публичная страница может обновиться не мгновенно из-за кеша чтения Supabase.

Если нужно мгновенное обновление после сохранения, это должно быть отдельно описано технически.

## Валидация

Админка должна проверять:

- обязательные текстовые поля не пустые;
- активная внешняя ссылка имеет URL;
- активная покупка имеет Telegram URL;
- цена курса хранится как текст;
- при создании курса название, описание и цена не пустые;
- описание курса не должно явно нарушать medical content rules;
- кнопка покупки остается `Купить`.

Медицинский смысл текста должен проверяться по `spec/feature-specs/medical-content-rules.md`.

## Безопасность

Запрещено:

- использовать service role key во frontend-коде;
- добавлять secret key в `NEXT_PUBLIC_*`;
- добавлять `DOCTOR_SUPABASE_SERVICE_ROLE_KEY` в browser/client Supabase;
- давать write-доступ роли `anon`;
- давать write-доступ любому `authenticated` пользователю без проверки `admin_users`;
- хранить пароль доктора в таблицах приложения;
- хранить ключи или токены в specs, work plans, roadmap или комментариях к коду.

## Проверка будущей реализации

Минимальные проверки после реализации:

```bash
npm run build
git diff --check
```

Ручные проверки:

- обычный посетитель открывает `/` без авторизации;
- обычный посетитель не может редактировать контент;
- доктор может войти через email/password;
- доктор может сохранить цену курса;
- доктор может создать новый курс с названием, описанием и ценой;
- новый активный курс появляется на публичной странице;
- новый неактивный курс не появляется на публичной странице;
- скрытый курс не оставляет пустое место на публичной странице;
- доктор может включить или выключить курс;
- доктор может изменить внешнюю ссылку;
- публичная страница показывает обновленные данные;
- публичный anon key не имеет write-доступа;
- service role key не используется во frontend.

## Порядок реализации после specs

1. Создать migration для `admin_users` и write policies.
2. Применить migration через Supabase CLI.
3. Проверить RLS и policies через MCP/CLI.
4. Настроить Supabase Auth email/password.
5. Создать пользователя доктора безопасным способом.
6. Реализовать `/admin` и форму входа.
7. Реализовать формы редактирования подтвержденных полей.
8. Проверить публичную страницу и отсутствие лишних функций.
