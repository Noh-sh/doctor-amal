# Technical Spec: Управление изменениями

## Цель
Описать безопасный порядок изменений в проекте Doctor-Amal после того, как первая версия уже реализована.

Этот документ нужен для ситуаций, когда проект завершен, прошло время, и нужно что-то изменить, добавить или исправить без поломки существующих сценариев.

## Главное правило
Любое изменение должно начинаться с понимания specs.

Нельзя менять код только по месту, если изменение затрагивает:
- пользовательский сценарий;
- модель данных;
- форму или валидацию;
- статус продаж;
- локальное хранение;
- медицинские формулировки;
- оплату;
- Telegram-доступ;
- будущий путь к онлайн-версии.

Если требования нет в specs, нельзя сразу писать код. Сначала нужно предупредить пользователя, получить подтверждение, обновить соответствующие specs и только после этого менять код.

Нельзя самостоятельно менять specs, чтобы легализовать новую функцию или новое поведение. Новое требование считается принятым только после подтверждения пользователя.

## Безопасный порядок работы
Перед изменением нужно:
1. Определить, какую часть проекта нужно изменить.
2. Найти связанные feature specs и technical specs.
3. Проверить, меняется ли поведение пользователя.
4. Проверить, есть ли требование в specs.
5. Если требования нет или оно неясное, остановиться и согласовать уточнение с пользователем.
6. Если поведение меняется, сначала обновить specs после подтверждения пользователя.
7. После обновления specs изменить код.
8. Проверить основные сценарии из `implementation-checklist.md`.
9. Проверить, что ограничения проекта не нарушены.

## Рабочий протокол перед реализацией
Перед обычной или рискованной реализацией нужно коротко зафиксировать:

```text
Проверка specs:
- Прочитано: ...
- Подтверждено: ...
- Вне specs: ...
- Решение: ...
```

Для мелких языковых и форматных исправлений без изменения смысла полный блок не нужен.

## Что можно делать без изменения specs
Specs можно не менять, если изменение:
- исправляет опечатки, орфографию, пунктуацию, грамматику или форматирование без изменения смысла;
- исправляет баг и возвращает код к уже описанному поведению;
- выбирает техническую реализацию уже описанного требования;
- уточняет структуру компонентов, helper-функции, repository/service слой или UI-детали без изменения пользовательского сценария, данных, маршрутов, валидации и бизнес-правил.

Если есть риск изменения смысла, сначала нужно спросить пользователя.

## Что требует согласования
Согласование пользователя требуется, если изменение:
- добавляет новое требование, функцию, сценарий, поле данных или маршрут;
- меняет пользовательское поведение, валидацию, статус продаж или business rule;
- добавляет интеграцию, оплату, авторизацию, аналитику, админку, серверный API или внешнюю запись данных;
- меняет медицинский смысл, условия покупки, доступ в Telegram или ожидания пользователя;
- переносит идею из `future-extension-plan.md` в текущую версию;
- вызывает сомнение, входит ли оно в specs.

После согласования нового требования сначала обновляются specs, потом код.

## Конфликты specs
Если specs прямо противоречат друг другу, нельзя выбирать вариант самостоятельно. Нужно остановиться, назвать конфликтующие файлы и спросить пользователя, какой вариант считать правильным.

Если specs дополняют друг друга, использовать каждый документ по его роли:
- `global-spec.md` и `functional-map.md` задают общие границы проекта;
- `user-stories/` описывают цели пользователя и пользовательский путь;
- `feature-specs/` описывают поведение конкретной функции;
- `technical-specs/` описывают техническую реализацию;
- `future-extension-plan.md` описывает будущее направление и не является текущим backlog.

## Future-extension plan
`future-extension-plan.md` можно использовать для понимания будущего направления и проверки совместимости решений.

Нельзя реализовывать функции из `future-extension-plan.md` в текущей версии без отдельного подтверждения пользователя и обновления актуальных specs.

## Работа с черновиками specs
AI может писать черновики specs, предлагать улучшения, искать противоречия, серые зоны и риски.

Любые изменения требований считаются принятыми только после подтверждения пользователя.

Если пользователь просит оценить spec, нужно дать оценку по 10-балльной шкале, объяснить причину оценки, что мешает 10/10 и что стоит добавить, убрать или уточнить.

## Рискованные изменения
Изменение считается рискованным, если оно:
- добавляет новый тип заявки;
- меняет модель `Course`, `Article` или `UserRequest`;
- меняет правила доступности покупки;
- меняет статусы продаж;
- меняет валидацию формы;
- добавляет сбор новых персональных данных;
- добавляет внешнюю интеграцию;
- добавляет серверную запись данных;
- добавляет админку;
- добавляет оплату или любые платежные поля;
- меняет медицинские формулировки;
- меняет способ доступа в Telegram.

Для рискованного изменения specs нужно обновить до изменения кода.

## Что AI не должен делать без проверки specs
AI не должен:
- добавлять онлайн-оплату;
- добавлять поля банковской карты;
- добавлять диагнозы как структурированные данные;
- превращать заявку в полноценную медицинскую консультацию;
- обещать автоматический доступ в Telegram;
- показывать черновики пользователю;
- менять статусы продаж только в UI без изменения domain rules;
- читать или писать данные напрямую из компонентов, если уже есть repository layer;
- удалять существующие ограничения проекта;
- менять пользовательский сценарий без обновления соответствующего spec.

## Regression checklist после изменений
После любого заметного изменения нужно проверить:
- главная страница открывается;
- страница "О докторе" открывается;
- каталог курсов открывается;
- поиск и фильтры курсов работают, если они затронуты;
- страница курса открывается по валидному `courseId`;
- несуществующий курс показывает понятную ошибку;
- заявка на покупку работает только при открытых продажах;
- заявка не доступна при закрытых продажах;
- форма показывает ошибки валидации;
- успешная заявка сохраняется;
- повторная заявка обрабатывается корректно;
- список статей открывается;
- страница статьи открывается;
- несуществующая статья показывает понятную ошибку;
- 404-страница работает;
- пользователь не видит технические ошибки;
- пользователь не видит служебные поля.

## Когда нужно обновить specs
Specs нужно обновить, если изменение:
- меняет текст или смысл пользовательского сценария;
- добавляет новую страницу;
- добавляет новый route;
- добавляет новое поле данных;
- меняет обязательность поля;
- меняет validation rule;
- меняет статус или business rule;
- меняет способ хранения;
- добавляет API;
- добавляет интеграцию;
- добавляет админскую функцию.

Если изменение только исправляет баг без изменения поведения, specs можно не менять. Но после исправления нужно пройти regression checklist для затронутой области.

## Как фиксировать изменения в specs
При изменении specs нужно:
- обновить feature spec, если меняется пользовательское поведение;
- обновить technical spec, если меняется реализация, модель данных, маршрут, хранение или валидация;
- обновить `implementation-checklist.md`, если появляется новый обязательный сценарий проверки;
- обновить `future-extension-plan.md`, если меняется путь к будущей онлайн-версии.

## Итоговое правило
Безопасный возврат к проекту после паузы строится так:

```text
понять задачу -> найти specs -> обновить specs при необходимости -> изменить код -> проверить regression checklist
```

Такой порядок защищает проект от случайных изменений, которые работают в одном месте, но ломают данные, заявки, продажи, медицинские ограничения или будущую онлайн-архитектуру.
