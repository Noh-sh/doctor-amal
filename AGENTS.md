# Правила работы с проектом Doctor Amal

## Язык проекта

- Все материалы проекта должны быть на русском языке.
- Это правило относится к спецификациям, документации, комментариям, пользовательским текстам интерфейса и рабочим пояснениям по проекту.
- Имена файлов, технические ключи, API-поля, команды, зависимости и синтаксис кода могут оставаться на английском, если этого требует экосистема или уже существующая структура проекта.
- Названия рабочих папок могут быть на русском или английском языке по решению владельца проекта. Это не считается нарушением правила о русском языке материалов проекта.

## Границы спецификаций

- Источник правды по требованиям находится в папке `spec/`.
- Для поиска релевантных требований использовать repo skill `doctor-amal-specs`, если он доступен.
- Нельзя реализовывать функциональность, сценарии, UI, данные или поведение, которых нет в спецификациях.
- Перед изменениями нужно сверить задачу с релевантными файлами из `spec/`.
- Если пользователь просит сделать то, чего нет в спецификациях, нужно сразу предупредить, что такого требования в `spec/` нет, и не реализовывать это без отдельного подтверждения.
- Если пользователь подтверждает новое требование, которого нет в `spec/`, сначала нужно обновить соответствующие specs, и только после этого реализовывать код.
- Если агент считает полезным добавить что-то сверх `spec/`, он обязан сначала предупредить пользователя, явно назвать, что именно выходит за границы specs, и получить согласование. Самостоятельно добавлять это в specs или код нельзя.
- Если требование найдено частично, нужно явно назвать, какая часть подтверждена спецификациями, а какая выходит за их границы.
- Нельзя самостоятельно менять продуктовые specs в `spec/`, чтобы легализовать новую функцию или новое поведение.
- Если specs противоречат друг другу, нужно остановиться, показать конфликт и спросить пользователя, какой вариант считать правильным.

## Что можно делать без отдельного изменения specs

- Исправлять опечатки, орфографию, пунктуацию, грамматику и форматирование, если смысл не меняется.
- Делать bugfix, если он приводит код к уже описанному в specs поведению.
- Принимать технические решения по реализации, если они нужны для выполнения specs и не добавляют новое продуктовое поведение.
- Выбирать UI-детали, отступы, состояния, структуру компонентов и техническую архитектуру, если они не меняют сценарий, смысл, данные, маршруты, валидацию или бизнес-правила.

## Что требует согласования

- Новые требования, функции, сценарии, поля данных, маршруты, интеграции, платежи, авторизация, аналитика, админка, серверный API и любые изменения бизнес-правил.
- Изменения пользовательских текстов, если они меняют обещание, медицинский смысл, условия покупки, статус продаж, доступ в Telegram или ожидания пользователя.
- Перенос будущих идей из `future-extension-plan.md` в текущую версию.
- Любое изменение, если есть сомнение, входит ли оно в specs.

## Работа со specs

- Агент может писать черновики specs, предлагать улучшения, искать противоречия, серые зоны и риски.
- Любые изменения требований считаются принятыми только после подтверждения пользователя.
- Если пользователь просит оценить spec, нужно дать оценку по 10-балльной шкале, объяснить причину оценки, что мешает 10/10 и что стоит добавить, убрать или уточнить.

## Проверка и git

- После изменений нужно выполнить проверку, соответствующую риску изменения. Если проверку выполнить нельзя, нужно сообщить об этом.
- Git-действия не выполнять без прямой просьбы пользователя. По умолчанию нужно дать пользователю команды и список файлов для ручного выполнения git.

## Рабочие планы

- Все этапы работы фиксируются в папке `Work plans/`.
- Активные задачи хранятся в `Work plans/Активные/`.
- Завершенные и проверенные задачи переносятся в `Work plans/Завершенные/`.
- Перед началом новой части работы нужно создать или обновить активный план.
- После каждой части работы нужно выполнить проверку, соответствующую риску изменения.
- Результат проверки нужно записать в файл плана.
- После завершения плана нужно обновить хронологию в папке `Roadmap/`.
- После завершения плана нужно обновить общую дорожную карту `Roadmap/project-roadmap.md`.
- Git commit выполняется только после проверки и подтверждения пользователя.
- Push на GitHub выполняется только после отдельного подтверждения пользователя.

## Порядок работы агента

Для любой новой задачи нужно идти в таком порядке:

1. Прочитать `AGENTS.md`.
2. Найти релевантные specs в `spec/` через skill `doctor-amal-specs`, если он доступен.
3. Проверить, что задача подтверждена specs и не выходит за текущие продуктовые границы.
4. Создать или обновить активный план в `Work plans/Активные/`.
5. Выполнить работу минимальным изменением, достаточным для подтвержденного требования.
6. Выполнить проверку, соответствующую риску изменения.
7. Записать результат проверки в файл плана.
8. Перенести завершенный и проверенный план в `Work plans/Завершенные/`.
9. Обновить `Roadmap/chronology.md` как changelog проекта.
10. Обновить `Roadmap/project-roadmap.md`.
11. Выполнить git commit только после проверки и прямого подтверждения пользователя.
12. Выполнить push только после отдельного подтверждения пользователя.

Если задача не требует изменения файлов, активный план не обязателен, но ответ все равно должен опираться на specs и текущий статус проекта.

## Именование новых файлов и папок

- Новые технические папки по возможности называть маленькими буквами, без пробелов, с разделением слов через дефис: например `work-plans`, `admin-tools`.
- Новые технические файлы по возможности называть в `camelCase` с маленькой первой буквой, без пробелов и дефисов: например `globalSpec.md`, `adminAuth.ts`, `userProfile.tsx`.
- Существующие исторические папки и файлы не переименовывать только ради стиля, если это не требуется отдельной задачей.
- Названия рабочих папок и документов на русском языке допустимы, если они уже являются частью текущей структуры проекта или явно выбраны владельцем проекта.

## Приоритеты

- Сначала соблюдать спецификации.
- Затем соблюдать существующую структуру проекта.
- Затем выбирать минимальное изменение, достаточное для выполнения подтвержденного требования.
