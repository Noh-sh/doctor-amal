# План 076: Защита от параллельного сохранения в админке

## Статус

Завершен.

## Цель

Исправить дефект надежности админки: пока одна форма сохраняется, нельзя запускать сохранение другой формы или повторную отправку через Enter.

## Подтверждение specs

Основание:

- `Work plans/Активные/068-master-plan-do-10-iz-10.md`;
- `spec/technical-specs/admin-auth-and-access.md`;
- `spec/technical-specs/supabase-content-source.md`;
- `spec/technical-specs/change-management.md`.

Подтверждено:

- админка должна давать понятный русский feedback;
- нужно проверить повторное сохранение и защиту от случайных двойных действий;
- bugfix можно делать без изменения specs, если он возвращает код к уже описанному поведению;
- менять бизнес-логику, данные, RLS или auth flow в этом плане не нужно.

Вне specs:

- новые поля, роли, policies, маршруты и admin-функции;
- изменение Supabase migrations;
- изменение текстов публичной страницы;
- новые сценарии покупки, заявки или оплаты.

## Что делаем

1. Заблокировать все кнопки сохранения, пока выполняется любое сохранение.
2. Добавить ранний выход в submit handlers, чтобы отправка через Enter не запускала параллельный запрос.
3. Не менять тексты, формы, Supabase helpers и бизнес-правила.
4. Выполнить quality gate.

## Проверка

Команды:

```bash
npm run quality
git diff --check
```

Ручная логика:

- во время сохранения одной формы остальные кнопки сохранения disabled;
- повторная отправка через Enter во время сохранения игнорируется;
- после завершения сохранения кнопки снова доступны.

## Критерии готовности

- Параллельные сохранения из UI заблокированы.
- Повторный submit во время сохранения не запускает новый запрос.
- `npm run quality` проходит.
- `git diff --check` проходит.

## Результат проверки

- В `AdminContentEditor` все кнопки сохранения получают `disabled`, пока выполняется любое сохранение.
- Каждый submit handler теперь делает ранний выход, если `savingKey` уже задан, поэтому повторная отправка через Enter не запускает параллельный запрос.
- Тексты, формы, Supabase helpers, RLS, migrations и бизнес-логика не менялись.
- `npm run quality` выполнен успешно.
- `git diff --check` выполнен без ошибок.
