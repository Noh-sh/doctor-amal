# План 056: Редактор контента в админке

## Статус

Завершен.

## Цель

Реализовать первый редактор контента внутри защищенного маршрута `/admin`, чтобы активный доктор-администратор мог менять подтвержденные текстовые данные страницы через Supabase.

Изменения после сохранения публикуются сразу и могут появиться на публичной странице после обновления данных и кеша.

## Подтверждение specs

Прочитано:

- `spec/feature-specs/admin-content-editing.md`
- `spec/user-stories/admin-user-stories.md`
- `spec/technical-specs/admin-auth-and-access.md`
- `spec/technical-specs/change-management.md`
- `spec/technical-specs/implementation-checklist.md`

Подтверждено:

- `/admin` уже является защищенным разделом для доктора;
- вход выполняется через email/password;
- редактировать контент может только активный `doctor_admin`;
- админка может менять только подтвержденные поля существующих строк Supabase;
- изменения публикуются сразу после сохранения;
- публичная страница `/` остается обычной Taplink-страницей без ссылки на админку;
- ошибки сохранения показываются понятным русским текстом без stack trace.

Вне specs и не делаем:

- не добавляем auth покупателей, регистрацию, личный кабинет, заявки, заказы, оплату, медицинские анкеты или аналитику;
- не добавляем роль менеджера;
- не добавляем черновики, предпросмотр публикации или журнал изменений;
- не редактируем `page_settings.medical_notice`;
- не добавляем загрузку или замену фото доктора;
- не добавляем `insert` или `delete` для курсов;
- не меняем Auth users, env-переменные, seed-файлы;
- не меняем RLS write policies;
- добавляем только технически необходимый `select`-доступ для активного `doctor_admin`, чтобы редактор мог загрузить существующие строки;
- не используем service role key во frontend-коде.

## Что делаем

1. Расширить защищенную оболочку админки после успешной проверки `doctor_admin`.
2. Загрузить из Supabase текущие данные:
   - `doctor_profile`;
   - `external_links`;
   - `courses`;
   - `purchase_settings`.
3. Добавить раздел редактирования профиля доктора:
   - краткое представление;
   - текстовые списки профиля;
   - без редактирования технического `id`;
   - без загрузки фото.
4. Добавить раздел редактирования курсов:
   - редактировать только существующие курсы;
   - `title`;
   - `description`;
   - `price_display_text`;
   - `price_is_confirmed`;
   - `is_active`;
   - без добавления, удаления, изменения `slug`, `purchase_label`, `sort_order` и технических id.
5. Добавить раздел редактирования внешних ссылок:
   - `url`;
   - `is_enabled`;
   - `inactive_text`;
   - только для существующих платформ `Telegram`, `WhatsApp`, `YouTube`, `Instagram`;
   - без добавления новых платформ.
6. Добавить раздел настройки покупки:
   - `purchase_settings.manager_telegram_url`;
   - `purchase_settings.inactive_text`;
   - без заказов, оплаты и автоматической передачи названия курса.
7. Сохранять изменения через обычный Supabase browser client под сессией доктора.
8. После сохранения обновлять данные в форме из Supabase.
9. Показывать состояния:
   - загрузка;
   - сохранение;
   - успешное сохранение;
   - понятная ошибка.

## Валидация

Минимально:

- обязательные текстовые поля не должны сохраняться пустыми;
- активная внешняя ссылка должна иметь URL;
- активная покупка должна иметь Telegram URL;
- цена курса сохраняется как текст;
- кнопка курса остается `Купить`;
- пользовательские ошибки показываются по-русски.

Медицинский смысл текстов админка технически не может полностью проверить автоматически. В UI можно показать короткое предупреждение о недопустимости обещаний лечения, но без отдельного workflow модерации.

## Техническая реализация

Предварительно:

```text
components/admin/AdminAccess.tsx
components/admin/AdminContentEditor.tsx
lib/supabase/adminContent.ts
styles/globals.css
```

Если при реализации будет проще оставить часть логики в одном client component, это допустимо, если поведение не меняется и код остается читаемым.

Новые зависимости не планируются: `@supabase/supabase-js` уже установлен на предыдущем этапе.

При реализации выявлено техническое условие: существующих `update` policies недостаточно для редактора, потому что доктор должен видеть текущие значения, включая неактивные курсы. Поэтому добавляется отдельная migration только для `select` policies активного `doctor_admin` на уже разрешенные контентные таблицы.

## UI-требования

- Все пользовательские тексты на русском языке.
- Интерфейс должен выглядеть как рабочая админка, не как landing page.
- Разделы должны быть компактными и понятными:
  - `Профиль`;
  - `Курсы`;
  - `Внешние ссылки`;
  - `Покупка`.
- Сохранение лучше делать по разделам или по отдельным сущностям, чтобы доктор понимал, что именно изменил.
- Публичная страница `/` не получает кнопку или ссылку на `/admin`.

## Безопасность

- Использовать только публичный Supabase key из `DOCTOR_SUPABASE_PUBLISHABLE_KEY` или legacy `DOCTOR_SUPABASE_ANON_KEY`.
- Не использовать `DOCTOR_SUPABASE_SERVICE_ROLE_KEY` во frontend-коде.
- Не добавлять секреты в `NEXT_PUBLIC_*`.
- Запись должна идти только через RLS под текущей сессией доктора.
- Пользователь без активной роли `doctor_admin` не должен видеть формы редактирования.
- Не хранить пароль доктора в коде, env, specs, work plans, roadmap или git.


Ручная проверка:

- `/` открывается без авторизации;
- `/` не показывает ссылку на `/admin`;
- `/admin` без сессии показывает вход;
- доктор входит через email/password;
- активный `doctor_admin` видит формы редактора;
- пользователь без роли `doctor_admin` не видит формы редактора;
- доктор может сохранить профильные тексты;
- доктор может сохранить цену курса;
- доктор может включить или выключить курс;
- доктор может изменить внешнюю ссылку;
- доктор может изменить Telegram-ссылку менеджера;
- публичная страница показывает обновленные данные после обновления данных и кеша;
- service role key не используется во frontend-коде;
- `page_settings.medical_notice` не редактируется;
- курсы не добавляются и не удаляются.

Ручные проверки с реальными изменениями нужно выполнять осторожно, потому что сохранение сразу меняет публичный контент Supabase.

## Критерии готовности

- В защищенной админке есть редактор подтвержденных полей.
- Все сохранения работают через Supabase и RLS.
- После сохранения форма обновляет данные.
- Ошибки понятны пользователю.
- Публичный сценарий `/` не изменен.
- Запрещенные функции не добавлены.
- Build проходит.
- Результат проверки записан в план.
