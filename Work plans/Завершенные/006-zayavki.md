# План 006: Заявки

## Статус

Завершен.

## Цель

Реализовать страницу заявки `/request` с двумя режимами: покупка выбранного курса и консультация по подбору курса. Форма должна проверять данные, сохранять заявку в browser `localStorage`, блокировать недоступные сценарии и показывать понятные сообщения без онлайн-оплаты и без обещаний медицинской консультации.

## Подтверждение specs

Основание:

- `spec/feature-specs/requests.md`
- `spec/technical-specs/requests-and-validation.md`
- `spec/technical-specs/local-storage.md`
- `spec/technical-specs/data-model.md`
- `spec/feature-specs/error-handling.md`
- `spec/technical-specs/routing-and-ui.md`
- `spec/user-stories/core-user-stories.md`
- `spec/user-stories/edge-case-stories.md`

Подтверждено:

- форма заявки поддерживает режимы `course_purchase` и `pre_purchase_consultation`;
- `/request?type=course_purchase&courseId=<courseId>` должен найти опубликованный курс, проверить открытые продажи и предзаполнить выбранный курс;
- `/request?type=pre_purchase_consultation` должен принимать заявку только если есть хотя бы один курс с открытыми продажами;
- обязательны имя, контакт, согласие с тем, что заявка не является медицинской консультацией, и согласие с тем, что оплата не выполняется на сайте;
- заявка на покупку требует выбранный опубликованный курс с `salesStatus = "open"`;
- заявка на консультацию требует возраст от 12 до 120 и достаточно подробное описание проблемы;
- контакт должен быть похож на email, Telegram username или телефон;
- заявки сохраняются в browser `localStorage` по ключу `doctor-amal:requests`, версия хранения `doctor-amal:storage-version = 1`;
- похожая заявка за последние 10 минут должна блокироваться или требовать явного повторного действия;
- при ошибке сохранения форма не очищается и не показывает ложный успех;
- UI не должен просить платежные данные, предлагать онлайн-оплату, обещать автоматический доступ в Telegram или создавать ощущение медицинской консультации внутри сайта.

Вне specs:

- онлайн-оплата;
- серверная отправка заявки;
- админка заявок;
- уведомления доктору или ассистенту;
- автоматическое добавление в Telegram;
- личный кабинет;
- сбор документов, анализов, платежных данных или файлов.

## Что сделано

1. Добавлен client-side repository/service для чтения и сохранения заявок в `localStorage`.
2. Добавлена валидация формы заявки по rules из specs.
3. Реализован client component формы заявки с режимами покупки и консультации.
4. Страница `/request` подключена к query params, опубликованным курсам и правилам открытых продаж.
5. Добавлены сообщения успеха, ошибки полей, блокировка недоступных сценариев и защита от двойной отправки.
6. Добавлены стили формы и состояний.
7. Результат отправки показывается рядом с кнопкой отправки, страница прокручивается к сообщению и переводит на него фокус.
8. В `/courses` обновлен общий блок статуса продаж: он показывает фактическое состояние продаж и соответствующие курсы.

## Что не делали в этом плане

- не отправляли заявки на сервер;
- не добавляли админку, экспорт или статусы обработки для владельца проекта;
- не меняли specs и бизнес-правила;
- не добавляли новые типы заявок;
- не реализовывали статьи, главную или страницу "О докторе" сверх уже существующих переходов.

## Проверка

Команды:

```bash
npm run build
```

Ручная проверка:

- `/request?type=course_purchase&courseId=skin-health-basics` показывает выбранный курс и позволяет отправить валидную заявку;
- повторная отправка похожей заявки за 10 минут блокируется понятным сообщением;
- `/request?type=course_purchase&courseId=nutrition-energy-support` блокирует покупку при закрытых продажах;
- `/request?type=pre_purchase_consultation` доступен, если есть хотя бы один курс с открытыми продажами;
- пустая форма показывает ошибки рядом с полями и summary;
- некорректный контакт не принимается;
- короткое описание проблемы для консультации не принимается;
- при ошибке `localStorage` не показывается сообщение об успехе;
- после отправки результат сразу виден рядом с кнопкой отправки;
- `/courses` показывает один общий блок фактического статуса продаж.

## Критерии готовности

- форма заявки работает в двух подтвержденных режимах;
- данные заявки соответствуют `UserRequest`;
- `localStorage` используется через repository/service, а не напрямую из страницы;
- недоступные сценарии продаж не позволяют отправить заявку;
- пользовательские тексты на русском языке и не выходят за медицинские/платежные ограничения specs;
- `npm run build` выполнен успешно;
- ручная проверка пользователем подтверждена.
