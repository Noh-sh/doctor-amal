# План 067: Актуализация и уточнение specs

## Статус

Завершен.

## Цель

Привести specs Doctor Amal к текущему фактически реализованному состоянию, убрать неоднозначные варианты из актуальных требований и четко отделить будущие функции от текущих.

## Подтверждение specs

Основание:

- пользователь подтвердил актуализацию specs без изменения структуры проекта;
- `spec/technical-specs/change-management.md`;
- фактический код публичной страницы и `/admin`;
- актуальные Supabase migrations.

Подтверждено фактической реализацией:

- публичный маршрут — `/`;
- защищенный маршрут админки — `/admin`;
- блоки `О докторе` и `Курсы` раскрываются через accordion;
- каждый курс раскрывается через accordion;
- внешние ссылки и `Купить` открываются в новой вкладке;
- Supabase является текущим online-источником контента с локальным fallback;
- имя доктора обязательно, остальные профильные текстовые поля необязательны;
- название, описание и цена курса обязательны;
- URL внешней ссылки обязателен только для активной ссылки;
- Telegram-ссылка менеджера необязательна; без нее нужен текст неактивной кнопки;
- `doctor_admin` может читать и обновлять разрешенный контент, а также создавать курсы;
- физическое удаление курсов, загрузка фото, управление Auth users, ролями и env не входят в текущую версию.

Вне specs:

- новых функций и нового поведения нет.

## Что делаем

1. Обновить верхнеуровневые specs под текущую online-версию.
2. Зафиксировать однозначные UI-решения, маршруты, обязательность полей и права доступа.
3. Перенести будущие возможности из актуальных требований в отдельный future-раздел или обозначить их только как `не входит`.
4. Убрать устаревшие формулировки о том, что Supabase и админка еще не реализованы.
5. Проверить согласованность specs с кодом и migrations.

## Что не делаем в этом плане

- Не меняем код приложения.
- Не меняем Supabase migrations или remote database.
- Не добавляем загрузку фото, удаление курсов, новые роли, оплату или другие будущие функции.
- Не меняем пользовательские тексты и медицинский смысл.

## Проверка

Команды:

```bash
rg -n "будущая админка|будущий маршрут|предварительный маршрут|рекомендуемая таблица|DOCTOR_SUPABASE_PUBLISHABLE_KEY|DOCTOR_SUPABASE_ANON_KEY" spec
rg -n "accordion|новой вкладке|/admin|doctor_admin|обязательные поля|необязательные поля" spec
npm run build
git diff --check
```

Ручная проверка:

- актуальные specs не называют уже реализованные Supabase и админку будущими;
- текущие требования не разрешают несколько вариантов для уже выбранного поведения;
- будущие функции не выглядят как разрешение на реализацию;
- требования соответствуют коду и migrations.

## Критерии готовности

- Актуальная версия проекта однозначно описана как online Taplink-страница с Supabase и защищенной админкой.
- Для текущего поведения выбран один фактически реализованный вариант.
- Будущие функции отделены от текущих требований.
- Код и структура проекта не изменены.

## Результат проверки

- Фактическое поведение публичной страницы, `/admin`, Supabase data layer и migrations сверено со specs.
- Поиск устаревших формулировок о локальной версии, будущей админке, будущем MCP и альтернативных вариантах текущего поведения не нашел совпадений.
- Подтверждено наличие в specs точных маршрутов, accordion-поведения, открытия внешних ссылок в новой вкладке, обязательности полей и прав `doctor_admin`.
- `npm run build` выполнен успешно.
- `git diff --check` выполнен без ошибок.
