# План 053: Начать подготовку админки и Auth для редактирования контента

## Статус

Завершен.

## Цель

Подготовить требования и безопасный порядок работ для будущей админки, через которую доктор сможет самостоятельно редактировать подтвержденный контент страницы: тексты, курсы, цены, внешние ссылки и Telegram-ссылку менеджера.

Этот план начинает этап подготовки. Он не реализует админку, не включает Auth в коде и не меняет Supabase policies.

## Подтверждение specs

Прочитано:

- `spec/technical-specs/future-extension-plan.md`
- `spec/technical-specs/change-management.md`
- `spec/technical-specs/supabase-content-source.md`

Подтверждено:

- будущая версия 2 может включать CMS или собственную админку;
- цель будущей версии 2 — дать доктору возможность самостоятельно управлять контентом без редактирования файлов проекта разработчиком;
- возможны вход для доктора или доверенного администратора, управление профилем, внешними ссылками, курсами, ценами, Telegram-ссылкой менеджера, включение и выключение курсов или кнопок;
- Supabase уже используется как источник контента страницы;
- Auth, админка, server API и новые права доступа требуют отдельного подтверждения, specs и work plan до реализации.

Вне specs:

- конкретная реализация `/admin` еще не утверждена;
- Supabase Auth еще не настраивается;
- пользователи доктора еще не создаются;
- RLS policies на запись еще не меняются;
- UI админки еще не создается;
- онлайн-оплата, заявки, покупатели, медицинские анкеты, личный кабинет и аналитика не входят в этот этап.

## Что делаем

1. Описать требования к будущей админке и Auth.
2. Определить роли: доктор, доверенный администратор, обычный посетитель.
3. Определить список редактируемых сущностей:
   - профиль доктора;
   - фото доктора, если позже будет подтверждена необходимость редактирования фото через админку;
   - внешние ссылки;
   - курсы;
   - цены;
   - общая Telegram-ссылка менеджера;
   - медицинское предупреждение.
4. Определить, какие поля нельзя редактировать через админку.
5. Выбрать базовый способ входа: Supabase Auth email/password или magic link.
6. Описать требуемые RLS policies для чтения и записи.
7. Описать минимальный UI будущей админки без реализации.
8. Подготовить список specs, которые нужно создать или обновить перед кодом.

## Что не делаем в этом плане

- Не меняем код приложения.
- Не создаем маршрут `/admin`.
- Не подключаем `@supabase/supabase-js` для Auth.
- Не создаем Supabase Auth users.
- Не меняем RLS policies.
- Не меняем migration-файлы.
- Не меняем seed-контент.
- Не добавляем заявки, оплату, покупателей, медицинские данные или аналитику.
- Не даем доктору прямой доступ в Supabase Dashboard как утвержденное продуктовое решение.

## Предварительные решения для обсуждения

Вопросы, которые нужно подтвердить перед specs и кодом:

1. Кто будет редактировать контент: только доктор или еще помощник/менеджер?
2. Какой вход использовать: email/password или magic link?
3. Нужен ли отдельный черновик/публикация или изменения сразу появляются на сайте?
4. Нужно ли редактировать фото доктора на первом админ-этапе?
5. Нужен ли журнал изменений?
6. Нужно ли ограничить редактирование медицинского предупреждения, чтобы не менять медицинский смысл без проверки?

Подтвержденные предварительные решения:

- контент редактирует только доктор;
- вход — email/password;
- изменения публикуются сразу после сохранения;
- журнал изменений на первом этапе не нужен;
- медицинское предупреждение на первом этапе не редактируется через админку;
- фото доктора редактируется через админку только когда в этом появится необходимость; для первого минимального этапа это нужно описать как отдельную опцию, потому что загрузка/замена фото требует отдельной проверки хранения файлов, размера, формата и безопасности.
