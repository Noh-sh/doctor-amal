# План 055: UI входа в админку

## Статус

Завершен.

## Цель

Реализовать первый интерфейс админки: защищенный маршрут `/admin`, вход доктора по email/password через Supabase Auth, проверку активной роли `doctor_admin` через `admin_users` и выход из админки.

Этот план не реализует формы редактирования контента. После входа доктор видит только защищенную оболочку админки и состояние готовности к следующему этапу.

## Подтверждение specs

Прочитано:

- `spec/feature-specs/admin-content-editing.md`
- `spec/user-stories/admin-user-stories.md`
- `spec/technical-specs/admin-auth-and-access.md`
- `spec/technical-specs/change-management.md`
- `spec/technical-specs/implementation-checklist.md`

Подтверждено:

- будущая админка доступна через защищенный раздел `/admin`;
- вход на первом этапе — email/password;
- редактировать контент может только доктор;
- роль `doctor_admin` проверяется через отдельную таблицу `admin_users`;
- обычный посетитель не должен видеть формы редактирования;
- публичная страница `/` не должна показывать ссылку на админку как пользовательское действие;
- ошибки входа и доступа должны быть понятными и без stack trace;
- auth покупателей, заявки, оплата, медицинские анкеты, аналитика, черновики, журнал изменений и роль менеджера не входят.

Вне specs:

- формы редактирования профиля, курсов, цен и ссылок не реализуются;
- загрузка или редактирование фото не реализуется;
- добавление или удаление курсов не реализуется;
- новые Supabase migrations не планируются, если текущие RLS/Auth проверки уже достаточны;
- service role key не используется во frontend-коде;
- публичный сценарий `/` не меняется.

## Что делаем

1. Установить `@supabase/supabase-js`, если библиотека еще не подключена.
2. Добавить browser Supabase client для Auth и чтения `admin_users` под сессией пользователя.
3. Создать маршрут `/admin`.
4. Создать client component формы входа:
   - email;
   - password;
   - submit;
   - состояние загрузки;
   - понятная ошибка.
5. После успешного входа проверить, есть ли у пользователя активная запись `doctor_admin` в `admin_users`.
6. Если роли нет, показать отказ доступа и выполнить безопасный logout или не показывать админскую оболочку.
7. Если роль есть, показать защищенную оболочку админки:
   - заголовок;
   - статус входа;
   - сообщение, что редактирование контента будет следующим этапом;
   - кнопку выхода.
8. Не добавлять ссылку на `/admin` на публичную страницу `/`.
9. Проверить build и ручные сценарии входа/отказа.

## Что не делаем в этом плане

- Не создаем редактор профиля.
- Не создаем редактор курсов.
- Не создаем редактор цен.
- Не создаем редактор внешних ссылок.
- Не создаем редактор Telegram-ссылки менеджера.
- Не меняем публичную Taplink-страницу.
- Не добавляем auth покупателей или регистрацию пользователей.
- Не добавляем заявки, оплату, заказы, медицинские анкеты или аналитику.
- Не добавляем загрузку фото.
- Не используем service role key.
- Не записываем пароль доктора в код, env, specs, work plans, roadmap или git.

## UI-требования

- Все пользовательские тексты на русском языке.
- `/admin` должен выглядеть как рабочий инструмент, не как landing page.
- Форма входа должна быть компактной и понятной.
- Ошибки должны быть понятными:
  - неправильный email или пароль;
  - нет доступа администратора;
  - Supabase недоступен.
- Не показывать stack trace или технические детали пользователю.
- Кнопка выхода должна быть доступна после входа.

## Безопасность

- Использовать только публичный Supabase key из `DOCTOR_SUPABASE_PUBLISHABLE_KEY` или legacy `DOCTOR_SUPABASE_ANON_KEY`.
- Не использовать `DOCTOR_SUPABASE_SERVICE_ROLE_KEY` во frontend-коде.
- Не добавлять `NEXT_PUBLIC_*` secret.
- Проверка роли должна читать `admin_users` под сессией пользователя и полагаться на RLS.
- Если пользователь вошел, но не является активным `doctor_admin`, формы редактирования не показывать.
