# План действий до разработки Supabase

## Статус

Завершенный справочный план.

Этот документ описывает порядок работы перед подключением Supabase. Он не является реализацией Supabase и не меняет поведение приложения.

Документ закрыт вместе с планом `044-plan-supabase-content-source.md`. Последующие Supabase-этапы `045`-`051` уже выполнены и проверены отдельными завершенными планами.

## Цель Supabase-этапа

Перевести источник контента Taplink-страницы с локального файла на Supabase, сохранив тот же публичный сценарий:

- одна страница `/`;
- блок `О докторе`;
- блок `Курсы`;
- внешние кнопки;
- кнопка `Купить` через Telegram менеджера;
- без форм, оплаты, личного кабинета и сбора пользовательских данных.

## Главный принцип

Supabase на первом этапе используется только как источник контента страницы.

Он не используется для:

- заявок;
- пользователей;
- авторизации покупателей;
- онлайн-оплаты;
- заказов;
- медицинских анкет;
- персональных медицинских данных;
- аналитики;
- автоматической выдачи доступа.

## Этап 2. Модель данных Supabase

На первом этапе достаточно таблиц:

1. `doctor_profile`
   - имя;
   - путь или URL фото;
   - короткое представление;
   - образование;
   - опыт;
   - профессиональные направления;
   - темы здоровья;
   - форматы помощи.

2. `external_links`
   - платформа;
   - label;
   - url;
   - active/inactive;
   - текст неактивного состояния;
   - порядок отображения.

3. `courses`
   - slug или технический id;
   - название;
   - описание;
   - цена текстом;
   - подтверждена цена или нет;
   - label кнопки покупки;
   - порядок отображения;
   - активен курс или нет.

4. `purchase_settings`
   - общая Telegram-ссылка менеджера;
   - текст неактивной покупки.

Дополнительно можно рассмотреть `page_settings`, если нужно хранить медицинское предупреждение в Supabase.

## Этап 3. Online Supabase Dashboard

В этом проекте первый Supabase-этап будет готовиться сразу в online Supabase Dashboard.

Локальный Supabase через Docker и Supabase CLI на этом этапе не используется.

Нужно подготовить:

- Supabase account;
- новый Supabase project;
- таблицы через SQL Editor или Table Editor;
- Row Level Security policies;
- тестовые записи контента;
- Project URL;
- publishable key или legacy anon key;
- `.env.local` в Next.js проекте.

Что делаем в Supabase Dashboard:

1. Создаем проект.
2. Создаем таблицы первого этапа.
3. Включаем RLS для таблиц.
4. Добавляем read-only policies для публичного чтения опубликованного контента.
5. Добавляем тестовый контент.
6. Проверяем, что публичные данные читаются только так, как разрешено policies.

Что не делаем:

- не создаем auth для покупателей;
- не создаем админку;
- не создаем платежи;
- не храним заявки;
- не храним пользовательские медицинские данные;
- не используем secret/service role key в браузере.

## Этап 4. Ключи и env-переменные

Ключи добавляются после создания online Supabase project и до написания Supabase client в коде.

Где взять:

- Supabase Dashboard;
- Project Settings;
- API Keys или Data API, в зависимости от текущего интерфейса Supabase.

Что нужно для публичного чтения контента:

```env
DOCTOR_SUPABASE_URL=...
DOCTOR_SUPABASE_PUBLISHABLE_KEY=...
```

Если в dashboard используется legacy key, вместо publishable key может быть:

```env
DOCTOR_SUPABASE_ANON_KEY=...
```

Правило для проекта:

- предпочтительно использовать publishable key, если он доступен в dashboard;
- legacy anon key допустим, если Supabase project пока показывает legacy keys;
- `.env.local` создается локально и не коммитится;
- для deploy эти же переменные добавляются в настройках hosting;
- secret key или service role key не добавлять в `NEXT_PUBLIC_*`;
- secret/service role key не нужен для первого этапа публичного чтения контента;
- если позже появится server-only задача, secret key описывается отдельным spec и хранится только как server-side env, например `DOCTOR_SUPABASE_SERVICE_ROLE_KEY`.

Пример для кода нужно будет выбрать в specs:

```ts
const supabaseUrl = process.env.DOCTOR_SUPABASE_URL;
const supabaseKey = process.env.DOCTOR_SUPABASE_PUBLISHABLE_KEY;
```

или legacy-вариант:

```ts
const supabaseKey = process.env.DOCTOR_SUPABASE_ANON_KEY;
```

На этом этапе нужно проверить:

- `.env.local` не попал в git;
- публичный ключ не дает лишних прав сверх RLS;
- secret/service role key не используется в frontend-коде;
- hosting env-переменные добавляются только после успешной локальной проверки сайта с online Supabase.

## Этап 5. Подключение Next.js к Supabase

После specs, online Supabase project и env-переменных можно переходить к коду.

Ожидаемые изменения:

- установить `@supabase/supabase-js`;
- добавить env-переменные в `.env.local`;
- создать Supabase client;
- создать слой получения данных;
- преобразовать rows из Supabase в текущий `TaplinkPageData`;
- сохранить UI-компоненты без переписывания;
- оставить проверки активности ссылок и кнопки `Купить`;
- обработать ошибки данных спокойным UI.

Главное правило: UI не должен знать, откуда пришли данные — из локального файла или Supabase.

## Этап 6. Проверки

Минимальные проверки после подключения:

```bash
npm run build
git diff --check
```

Ручные проверки:

- `/` открывается;
- неизвестные URL показывают 404;
- блок `О докторе` раскрывается;
- блок `Курсы` раскрывается;
- курсы и цены приходят из Supabase;
- внешние кнопки без URL остаются неактивными;
- `Купить` без Telegram-ссылки остается неактивной;
- нет форм, оплаты, заявок и сбора пользовательских данных.

## Этап 7. Deploy

Так как Supabase project уже online, после локальной проверки Next.js нужно подготовить deploy.

Порядок:

1. Убедиться, что online Supabase таблицы и policies готовы.
2. Добавить `DOCTOR_SUPABASE_URL` и выбранный публичный ключ в hosting env.
3. Проверить production build.
4. Проверить страницу online.
5. Проверить, что секретные ключи не добавлены в frontend env.
