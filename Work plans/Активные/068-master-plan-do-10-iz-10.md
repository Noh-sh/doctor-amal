# План 068: Довести текущую версию до 10 из 10

## Статус

Активный мастер-план.

## Цель

Пошагово довести текущую подтвержденную версию Doctor Amal до production-уровня 10/10 без изменения ее продуктовых границ:

- публичная Taplink-страница `/`;
- защищенная админка `/admin`;
- Supabase как основной источник контента;
- локальный fallback публичной страницы;
- редактирование разрешенного контента активным `doctor_admin`;
- без заявок, auth покупателей, онлайн-оплаты и других неподтвержденных функций.

## Что означает 10/10

Оценка 10/10 относится к качеству текущего подтвержденного продукта, а не к количеству функций.

Проект считается готовым на 10/10, когда:

- актуальные specs, код, migrations и production совпадают;
- основные публичные и административные сценарии покрыты автоматизированными проверками;
- RLS и права доступа проверены отрицательными и положительными сценариями;
- публичная страница и админка проверены на мобильных и desktop-размерах;
- обработаны ошибки, загрузка, пустые и неактивные состояния;
- секреты не попадают в git или frontend bundle;
- production deploy проверен по приемочному checklist;
- есть понятный порядок deploy, rollback и восстановления доступа;
- рабочее дерево содержит только понятные и проверенные изменения;
- все проверки проходят перед финальной оценкой.

## Подтверждение specs

Основание:

- `spec/global-spec.md`;
- `spec/functional-map.md`;
- `spec/feature-specs/`;
- `spec/user-stories/`;
- `spec/technical-specs/architecture.md`;
- `spec/technical-specs/admin-auth-and-access.md`;
- `spec/technical-specs/change-management.md`;
- `spec/technical-specs/implementation-checklist.md`;
- `spec/technical-specs/future-extension-plan.md`;
- `Design map/design-rules.md`.

Подтверждено:

- можно исправлять bugs и повышать техническое качество без изменения пользовательского сценария;
- нужно проверять `/`, `/admin`, 404, accordion, внешние ссылки, состояния без URL, валидацию, Auth и RLS;
- migrations являются источником правды для структуры Supabase;
- MCP используется только для read-only инспекции и проверки;
- дизайн можно улучшать в рамках утвержденных правил без изменения сценариев, данных и маршрутов.

Вне текущих specs и не входит в обязательный путь до 10/10:

- загрузка и замена фото через админку;
- физическое удаление курсов;
- новые роли и управление Auth users через интерфейс;
- аналитика и UTM;
- черновики, audit log и предпросмотр публикации;
- заявки, auth покупателей, онлайн-оплата и автоматическая выдача доступа;
- новые публичные маршруты, поиск и фильтры.

Каждый такой пункт требует отдельного подтверждения и обновления specs до реализации.

## Правило выполнения

Этот файл является мастер-планом. Каждый этап выполняется отдельным активным work plan.

Переход к следующему этапу разрешен только после:

1. выполнения проверок текущего этапа;
2. записи результата в его work plan;
3. обновления Roadmap;
4. подтверждения пользователя перед commit;
5. отдельного подтверждения пользователя перед push.

## Этап 0. Зафиксировать чистую базовую точку

Цель: начать повышение качества от понятного и воспроизводимого состояния.

Действия:

1. Проверить итоговый diff плана `067`.
2. Повторно выполнить `npm run build` и `git diff --check`.
3. Проверить отсутствие секретов и `.env.local` в git.
4. По прямой просьбе пользователя создать отдельный commit актуализированных specs.
5. По отдельной просьбе пользователя выполнить push.
6. Убедиться, что рабочее дерево чистое перед следующим этапом.

Критерий завершения:

- актуальные specs находятся в отдельном проверенном commit;
- локальная ветка и remote имеют понятное состояние;
- секреты не закоммичены.

## Этап 1. Провести полный технический аудит текущей версии

Цель: получить точный список дефектов и рисков до изменения кода.

Действия:

1. Сверить код с `implementation-checklist.md`.
2. Проверить архитектуру публичной страницы, data layer и админки.
3. Проверить зависимости, scripts и доступные статические проверки.
4. Проверить обработку loading, error, 404 и fallback.
5. Проверить отсутствие неподтвержденных маршрутов, форм и сбора данных.
6. Проверить тексты на соответствие medical content rules.
7. Составить findings по severity с файлами и строками.
8. Создать отдельные bugfix-планы только для подтвержденных проблем.

Проверка:

```bash
npm run build
git diff --check
```

Критерий завершения:

- все найденные риски классифицированы;
- нет неизвестных критических или высоких дефектов;
- каждый подтвержденный дефект имеет отдельный план исправления.

## Этап 2. Добавить автоматизированный quality gate

Цель: сделать регрессии видимыми до deploy.

Требует отдельного технического плана, потому что в проекте сейчас нет test runner и E2E-инфраструктуры.

Действия:

1. Выбрать минимальный test stack, совместимый с текущим Next.js.
2. Добавить рабочие команды статической проверки, typecheck и тестов.
3. Добавить unit-тесты для чистых функций и преобразования данных.
4. Добавить E2E-тесты публичных сценариев:
   - `/`;
   - 404;
   - accordion `О докторе`;
   - accordion `Курсы`;
   - раскрытие курса;
   - активные и неактивные внешние ссылки;
   - активная и неактивная кнопка `Купить`;
   - отсутствие форм и неподтвержденных маршрутов.
5. Добавить E2E-тесты `/admin`, не раскрывая реальные credentials в git.
6. Добавить единый локальный quality-check script.

Предварительный обязательный gate:

```bash
npm run typecheck
npm run lint
npm test
npm run test:e2e
npm run build
git diff --check
```

Критерий завершения:

- обязательный quality gate выполняется одной понятной последовательностью команд;
- критические публичные и административные сценарии автоматически проверяются;
- тестовые секреты не находятся в git.

## Этап 3. Проверить Supabase, Auth, RLS и секреты

Цель: доказать, что права доступа работают по принципу минимальных привилегий.

Действия:

1. Сверить remote migrations с `supabase/migrations/`.
2. Проверить RLS на всех текущих таблицах.
3. Проверить положительные сценарии активного `doctor_admin`.
4. Проверить отрицательные сценарии:
   - `anon` не пишет контент;
   - обычный `authenticated` без admin-доступа не пишет контент;
   - неактивный admin не пишет контент;
   - admin не изменяет запрещенные поля;
   - admin не выполняет `delete`;
   - frontend не использует service role key.
5. Проверить, что `.env.local` игнорируется и секреты отсутствуют в git history текущей ветки.
6. Документировать безопасное создание, отключение и восстановление доступа доктора без добавления управления пользователями в UI.
7. Проверить production env в Vercel вручную без записи значений ключей в документацию.

Критерий завершения:

- remote schema и migrations совпадают;
- RLS-проверки подтверждают разрешенные и запрещенные действия;
- service role key не попадает во frontend;
- процедура восстановления admin-доступа понятна владельцу проекта.

## Этап 4. Повысить надежность данных и ошибок

Цель: публичная страница остается полезной, а админка понятной при ожидаемых сбоях.

Действия:

1. Проверить Supabase success, empty, partial-data и unavailable scenarios.
2. Проверить локальный fallback публичной страницы.
3. Проверить валидацию обязательных и необязательных полей админки.
4. Проверить ошибки входа, отказ доступа, истекшую сессию и ошибки сохранения.
5. Проверить повторное сохранение и защиту от случайных двойных действий.
6. Проверить, что технические ошибки не показывают stack trace или секретные детали.
7. Исправить только bugs относительно текущих specs.

Критерий завершения:

- публичная страница не ломается при недоступности Supabase;
- админ получает понятный русский feedback;
- ошибки не раскрывают технические детали;
- все сценарии покрыты автоматизированными или документированными ручными проверками.

## Этап 5. Провести UI, responsive и accessibility-приемку

Цель: текущая публичная страница и админка выглядят и работают профессионально на поддерживаемых экранах.

Действия:

1. Проверить публичную страницу и `/admin` на ширинах `360`, `390`, `768`, `1024`, `1440`.
2. Проверить отсутствие переполнения, перекрытий и горизонтального скролла.
3. Проверить клавиатурную навигацию, focus states, labels, `alt`, контраст и disabled-состояния.
4. Проверить длинные реальные значения контента.
5. Проверить loading, error, empty, disabled и success states.
6. Улучшить дизайн публичной страницы и админки только в рамках текущих specs и `Design map/design-rules.md`.
7. Зафиксировать визуальные результаты скриншотами.

Критерий завершения:

- все проверочные размеры проходят приемку;
- интерфейс доступен с клавиатуры;
- текст и элементы не перекрываются;
- дизайн публичной страницы и админки согласован и не меняет продуктовый сценарий.

## Этап 6. Подготовить production и эксплуатацию

Цель: deploy и восстановление выполняются предсказуемо.

Действия:

1. Проверить production build из чистого состояния.
2. Проверить production URL:
   - `/`;
   - `/admin`;
   - 404;
   - чтение Supabase;
   - вход доктора;
   - сохранение разрешенного контента;
   - обновление публичной страницы после `revalidate: 30`.
3. Проверить Vercel env scopes для Production и Preview.
4. Описать короткий deploy checklist.
5. Описать rollback к последнему рабочему deploy.
6. Описать восстановление admin-доступа.
7. Проверить отсутствие секретов в logs, browser bundle и репозитории.

Опционально и только после отдельного подтверждения specs:

- внешняя error-monitoring интеграция;
- uptime monitoring;
- аналитика.

Критерий завершения:

- production-проверка пройдена;
- deploy, rollback и восстановление доступа документированы;
- владелец проекта может повторить основные эксплуатационные действия.

## Этап 7. Финальная приемка 10/10

Цель: подтвердить оценку фактами.

Действия:

1. Повторно пройти весь `implementation-checklist.md`.
2. Выполнить полный автоматизированный quality gate.
3. Повторно проверить RLS и production.
4. Убедиться, что нет открытых critical/high findings.
5. Убедиться, что каждый medium/low finding либо исправлен, либо явно принят владельцем проекта.
6. Проверить чистоту git и соответствие Roadmap.
7. Заполнить финальную таблицу оценки по всем областям.

Критерий завершения:

- каждая область получила полный балл;
- нет известных неподтвержденных отклонений от specs;
- все проверки записаны в завершенном плане;
- пользователь подтвердил финальную приемку.

## Порядок ближайших действий

1. Завершить этап 0: проверить и зафиксировать изменения плана `067`.
2. Создать отдельный активный план этапа 1: полный технический аудит.
3. После аудита выполнять отдельные этапы и bugfix-планы по одному.
4. Не начинать будущие функции из `future-extension-plan.md` в рамках этого мастер-плана.

## Что не делаем в этом мастер-плане

- Не реализуем все этапы одним большим изменением.
- Не смешиваем quality work с новыми продуктовыми функциями.
- Не меняем remote Supabase напрямую вне migrations.
- Не коммитим секреты.
- Не выполняем commit без подтверждения пользователя.
- Не выполняем push без отдельного подтверждения пользователя.

## Проверка мастер-плана

Ручная проверка:

- этапы идут от базовой точки к аудиту, автоматизации, безопасности, надежности, UI и production;
- каждый этап имеет измеримый критерий завершения;
- будущие функции отделены от обязательного пути до 10/10;

## Критерии готовности мастер-плана

- определена измеримая шкала оценки;
- первый следующий шаг однозначен;
- новые функции не смешаны с повышением качества текущей версии.

## Результат проверки

- План сверено с актуальными specs, `implementation-checklist.md`, `future-extension-plan.md` и `Design map/design-rules.md`.
- Будущие функции явно отделены от обязательного пути до 10/10.
